## [Unreleased ‚Äî 0.3.0-dev]

### Added
- **Short IDs + `last` shortcut everywhere (UX-2)** ‚Äî `octo sessions list`, REPL `/sessions`, and `octo task list` now show the 8-character hex suffix of each ID as the leftmost column (with a human-readable created-at next to it), so the user copies 8 characters instead of 23. Every CLI command that consumes an ID ‚Äî `octo chat -c <id>`, `octo task status/show/resume/cancel/run` ‚Äî accepts: the full ID (unchanged), the 8-char short ID, any unique substring of an ID, or the literal `last` to mean "most recent". Ambiguity is an error that lists candidates so the user can re-disambiguate. The disk format is untouched ‚Äî files are still `YYYYMMDD-HHMMSS-<hex>.{jsonl,json}`, which keeps lexicographic-sort = chronological-sort and means zero migration for existing sessions/tasks. Same idea git uses: long SHA on disk, 7-char short SHA in `git log`.
- **REPL line editing + history (UX-1)** ‚Äî `octo chat` now drives the prompt through `chzyer/readline` when stdin is an interactive terminal: up/down arrow recall a persistent history at `~/.octo/history` (override via `OCTO_HISTORY_FILE`), Ctrl-A / Ctrl-E / Ctrl-W and the usual readline editing keys work, and a trailing `\` continues the input onto a `... ` continuation line (bash-style; doubled `\\` is a literal). At an idle prompt Ctrl-C clears the line instead of exiting (use `/exit` or Ctrl-D), matching `bash` / `zsh` convention. The asker (`ask_user_question`) and the permission gate share the same line reader, so the `Other (free text)` and `allow? [y/a/N]` prompts get history + editing too. Non-tty stdin (pipes, tests, `octo chat < input.txt`) keeps the existing scanner path verbatim ‚Äî no behaviour change there. First of five planned CLI-UX PRs; see `dev-docs/go-rewrite-roadmap.md` for the rest (ID shortcuts, error messages + `octo help`, status feedback + verbosity, shell completion).

## [0.2.0] ‚Äî 2026-05-28

Adds **M11** (autonomous task orchestration) and closes **C9 Phase 2 + Phase 3** (the memory daemon and the turn-boundary hook surface for retrieval layers like Hindsight).

### Added
- **C9 Phase 2 ‚Äî memory daemon** (#115, #116, #117) ‚Äî `octo memoryd start|stop|status` runs a long-lived foreground process that takes over the boundary extraction + consolidation passes the chat path runs at startup. Sessions are scanned each tick (15s default); a session whose mtime hasn't moved in 15 minutes is eligible for extraction, so the daemon never races a still-active chat. One session per tick by design ‚Äî `SIGTERM` never finds the daemon mid-extract for more than one session's worth of LLM time. Provider config read from env (`ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `OCTO_MEMORYD_PROVIDER` override); refuses to start without a key. Chat side detects the daemon via the PID file and skips the in-process memory pass, so REPL startup is effectively instant when the daemon is alive. Windows fail-soft (daemon refuses; chat keeps its Phase 1 path).
- **C9 Phase 3 ‚Äî turn-boundary hooks** (#118, #119) ‚Äî `internal/hooks` exposes pre-turn / post-turn shell hooks at the REPL turn boundary, the same surface Claude Code's `UserPromptSubmit` hook uses. The pre-turn hook receives `{"user_input": "..."}` on stdin and its stdout (text or `{"additional_context": "..."}`) is folded into the user message via the canonical divider. The post-turn hook receives `{user_input, assistant_reply}` for retention; output is ignored. Config via env vars (`OCTO_HOOK_PRE_TURN`, `OCTO_HOOK_POST_TURN`, `OCTO_HOOK_TIMEOUT`); zero env vars ‚Üí zero-cost no-op. 5s default timeout, 30s ceiling, `cmd.WaitDelay = 1s` so a child `sleep` can't keep the pipes open after `SIGKILL`. Pre-turn failures log but never block the turn; post-turn runs only on a successful turn so failed turns don't pollute the retention index. `dev-docs/c9-phase3-hindsight.md` walks through wiring Hindsight (and other retrieval layers) into this surface with copy-paste recall/retain scripts.
- **M11 ‚Äî autonomous task orchestration** (#110, #111, #112, #113) ‚Äî `octo task start "<goal>"` decomposes the goal into a DAG of subtasks via a planner side-call, then dispatches one M10 sub-agent per ready node in parallel, fsync'ing state per transition so a `kill -9` resumes cleanly. The harness drives task-graph state ‚Äî never sends the accumulated conversation to an evaluator (that's the failure mode of `/goal`-style designs on long tasks). Subcommands:
  - `octo task start "<goal>" [--plan-only]` ‚Äî plan + run end-to-end (or just plan).
  - `octo task run <id>` ‚Äî execute a previously planned task.
  - `octo task list` (alias `ls`) ‚Äî newest-first table of every task.
  - `octo task status <id>` ‚Äî full DAG state with per-subtask glyphs (‚ñ∂ ‚óã ‚úì ‚úó ‚äò), `blocked_by` lines, one-line result/error snippets.
  - `octo task show <id> <subtask-id>` ‚Äî full Description / Started / Finished / Duration / Error / Result for one subtask.
  - `octo task resume <id>` ‚Äî resets Failed / Skipped subtasks to Pending and re-runs (Done subtasks keep their results).
  - `octo task cancel <id>` ‚Äî marks a task Cancelled.

  New `internal/taskgraph` package owns the Task / Subtask data model + JSON-with-fsync persistence under `~/.octo/tasks/<id>.json`. `agent.PlanTask` is a new side-call with an 80-line planner system prompt (no-op gate, DAG anti-patterns, evidence‚Üíimplication shape borrowed from the stage_one memory extract). `taskgraph.Scheduler` walks the DAG with **stop-on-first-failure** semantics: a failed subtask marks the whole task Failed and stamps remaining Pending subtasks as Skipped, so `resume` can distinguish "didn't try" from "tried and failed". Foreground execution only in v1 (no daemon). See `dev-docs/go-rewrite-roadmap.md` ¬ßM11 for design + acceptance criteria.

## [0.1.0] ‚Äî 2026-05-28

First numbered Go release. Contains the entire `octo-agent` rewrite from the Ruby baseline plus 14 milestones of feature work (M2‚ÄìM7, M10, parity items C1‚ÄìC12 + C8/C9 typed memory, OS-level sandbox, identity files, ask_user_question, task_manager).

### Fixed
- **SSE parsing tolerant of `data:` without trailing space** (#72) ‚Äî strict `data: ` decoders missed events from some third-party servers that emit `data:<value>` (no space).
- **Delta-message cache usage parsing** (#72) ‚Äî Anthropic streams cache token counts in a final `message_delta` event; previously we only read those from `message_start`, so streaming sessions reported zero cache hits.
- **Session ID collisions** ‚Äî session IDs were `YYYYMMDD-HHMMSS` (second precision), so two sessions started in the same second silently overwrote each other's save file. IDs now carry an 8-hex-char (32-bit) random suffix (`YYYYMMDD-HHMMSS-xxxxxxxx`), keeping them sortable/readable while making same-second collisions negligible.

### Changed
- **History compaction default-on with window-relative threshold** (#75, parity C6) ‚Äî `--compact-threshold 0` (the default) now means "auto-trigger near `windowSize √ó 0.8`" computed from the model's context window, instead of "disabled". Negative values still disable; explicit positives override.
- **OpenAI reasoning_content round-trip** (#73) ‚Äî third-party reasoning-capable models (DeepSeek-R1, Qwen thinking models) emit `reasoning_content` alongside `content`. Both are captured and replayed back in subsequent turns so the model can refer to its own prior reasoning across tool calls.
- **Append-only JSONL session persistence** (parity C7) ‚Äî sessions are now stored as a JSONL transcript (`~/.octo/sessions/<id>.jsonl`: a meta line + one line per message) instead of a single JSON object rewritten in full on every turn. Each save **appends** only the messages added since the last save (per-turn cost O(new messages), not O(total history)), falling back to a full rewrite only when history shrinks (i.e. compaction). "Last updated" comes from file mtime, so `--list-sessions` no longer needs a stored `updated_at`. **Breaking:** old `.json` session files are not read (no migration); resume-by-id and the `.json`-suffix-tolerant lookup still work for `.jsonl`.
- **Structured compaction summary** (parity C3) ‚Äî the history-compaction side-call now uses a structured five-section summary prompt (Primary Request / Key Technical Concepts / Files and Code / Errors and Fixes / Current State and Next Steps) instead of a freeform "summarize this" instruction, mirroring Claude Code's compaction prompt ‚Äî a structured carry-forward retains far more usable signal.
- **Parallel dispatch of read-only tools** ‚Äî when the model emits several tool calls in one turn and they're all side-effect-free (`read_file` / `glob` / `grep` / `web_fetch` / `web_search`), `dispatchTools` now runs them concurrently instead of one-at-a-time, cutting latency on the common "read three files at once" pattern. The permission gate still runs serially first (an interactive `ask` prompt must not race another on stdin); only the allowed calls execute, and results stay in block order. Any mutating or streaming tool in the batch (or a single call) keeps the serial path, preserving `tool_progress` streaming.
- **Loop budget gates** ‚Äî the agentic loop's hardcoded 20-iteration cap (which `error`ed out and discarded the turn) is replaced by configurable `Agent.MaxTurns` (default 20) and `Agent.MaxCostUSD` (0 = unlimited), surfaced as `octo chat --max-turns` / `--max-cost`. Hitting either limit now ends the run *gracefully*: history keeps the partial progress and the caller gets a friendly assistant message (`[octo] Stopped: ‚Ä¶`) with a sentinel `StopReason` (`max_turns` / `max_cost`) instead of an error. The cost gate checks the running session estimate before each provider call. (Ports Ruby CATCHUP P0-4.)
- **Unified agentic loop** ‚Äî `Run` (buffered) and `RunStream` (streaming) previously carried two near-duplicate copies of the tool-dispatch loop, so changes like the permission gate had to be applied in both. They now share a single `runLoop`; `Run` drives it with a buffered send closure and a nil event handler (every event-emit path short-circuits on nil), `RunStream` with a streaming closure and a real handler. Behaviour is unchanged ‚Äî verified by the existing `Run` and `RunStream` test suites.

### Added
- **`task_manager` tool group** (#108) ‚Äî three tools (`task_create` / `task_update` / `task_list`) let the model break work into discrete trackable steps. Tasks carry an `id` (monotonic, never reused), `subject` (imperative title), `description`, `status` (`pending | in_progress | completed | deleted`), and `active_form` (present-continuous label surfaced when in-progress so the row reads like a spinner). Session-scoped in-memory store; cross-session persistence is M11 territory. `/tasks` REPL slash command renders the same list via the same formatter `task_list` returns to the LLM, so user-facing and model-facing rendering can't drift.
- **`ask_user_question` tool** (#107) ‚Äî model asks one structured clarifying question with 2-4 options and an automatic "Other (free text)" tail; supports `multi_select` and an optional `header` tag. REPL renders a numbered list, parses comma-separated picks for multi-select, returns `User chose: ‚Ä¶` (or `(user cancelled)`) as the tool_result. Gated REPL-only ‚Äî absent from `DefaultTools()` until an Asker is registered, so single-turn/unattended modes don't advertise a tool that can only error.
- **Sub-agent-backed memory consolidator** (#105) ‚Äî consolidation now spawns a sub-agent with a read-only allowlist (`read_file` / `grep` / `glob`) that can autonomously navigate the memory dir (grep `rollout_summaries/`, read individual `<slug>.md` files for context) and return the new summary text. Falls back to the existing `ConsolidateMemory` side-call when no Spawner is wired or the sub-agent errors. The parent stays in charge of `WriteSummary` + `ArchiveAll` so the v1 marker + git commits happen in one place.
- **M10 ‚Äî `launch_agent` sub-agent tool** (#104) ‚Äî model can spawn a child agent that handles a focused sub-task in isolation and reports a final reply. Child shares the parent's `Sender` (one provider connection) and `System` (same harness identity) but runs on a fresh `History`, capped at 12 turns so a runaway child can't burn budget unnoticed. Token usage rolls back into the parent's session totals via `Agent.AccrueChildUsage` so `/cost` reports one consolidated number. Multiple `launch_agent` calls in one `tool_use` batch fan out concurrently. Recursion guard: `launch_agent` is filtered out of the child's tool list AND a context marker is stamped so a hallucinating child can't recurse.
- **C9 ‚Äî typed cross-session memory** (#93‚Äì#102) ‚Äî `internal/memory` package owns `~/.octo/memory/`. **Typed entries**: one fact per `<slug>.md` with YAML frontmatter (`name / description / type / created / last_verified`); `type` is `user | feedback | project | reference`. **`remember` tool** writes immediately when the model recognizes a load-bearing signal (preference, correction, project decision, external resource pointer). **Boundary extraction** runs as a side-call at REPL start over the previous session's transcript to sweep what the model didn't record live. **Incremental consolidate** folds new entries into a maintained `memory_summary.md` instead of rebuilding; `ArchiveAll` drops folded entries from the working tree (#96). **Three-piece per-rollout artifacts** (#102) ‚Äî each consolidation produces `rollout_slug` + `rollout_summary` + `facts[]`; the rollout summary lives in `rollout_summaries/<timestamp>-<slug>.md` as an on-demand reference that future sessions can grep. **Stage_one-style extract prompt** (#98) ‚Äî explicit no-op gate, outcome triage (success / partial / fail / uncertain), evidence‚Üíimplication wording for feedback, user > tool > assistant reading priority. **`memory_summary.md` v1 protocol marker** (#100) ‚Äî `<!-- octo-memory v1 -->` HTML comment as the first line; stripped on read so the LLM never sees it; HTML-comment so Markdown viewers render it invisibly. **Git baseline replaces `archive/` subdir** (#101) ‚Äî `~/.octo/memory/` is auto-`git init`ed on first write; every Save / WriteSummary / ArchiveAll commits, so git history is the archive. `Store.WorkspaceDiff(baseSHA)` + `LastConsolidatedSHA` are wired for the sub-agent consolidator. **base.md Memory section** expanded to ~60 lines with citations (`(from memory: ‚Ä¶)` inline attribution when a recalled fact is load-bearing), verify-first (grep/read remembered paths before acting), and "user contradicts memory ‚Üí user wins" (#99). REPL `/memory` + `octo memory list [--archive]` inspect the store; `--no-memory` disables. Design + Codex comparison: [`c9-memory-design.md`](dev-docs/c9-memory-design.md).
- **Identity files** (#92) ‚Äî `~/.octo/soul.md` (agent persona) and `~/.octo/user.md` (user profile) layered into the system prompt below `base.md` so the agent has a consistent voice and the user's name/role/preferences are background context. Full nine-layer composition: `base ‚Üí soul ‚Üí env ‚Üí skills ‚Üí memory ‚Üí user.md ‚Üí octorules(user) ‚Üí octorules(project) ‚Üí --system`. Design: [`identity-files-design.md`](dev-docs/identity-files-design.md).
- **M7 ‚Äî Skill loader** (#87) ‚Äî Claude Code-compatible `SKILL.md` format (YAML frontmatter + body). Skills discovered at session start from `~/.octo/skills/<name>/` (user) and `./.octo/skills/<name>/` (project) and rendered into a one-line-per-skill manifest in the system prompt (L1 "what's available"). The new `skill` tool serves full bodies on demand (L2 "give me the instructions"). REPL `/skills` lists them; `/<name>` triggers one directly. `--list-skills` enumerates from the CLI. Symlinks under `~/.claude/skills/` work directly (the format is bit-compatible).
- **C11 ‚Äî OS-level command sandbox** (#82, #84) ‚Äî opt-in `--sandbox` wraps every `terminal` invocation in macOS Seatbelt (`sbpl` policy) / Linux Landlock + seccomp (with `bwrap` fallback). Default policy: filesystem writes restricted to cwd, network disabled. `--sandbox-write-root` / `--sandbox-read-root` extend the policy with extra paths; `--sandbox-allow-net` toggles network on. Windows degrades to a clear notice. Stays opt-in by deliberate decision (#89) ‚Äî `c11-sandbox-design.md` ¬ß8 documents why default-on was not pursued, and Phase 2 host allowlist (#85) was evaluated and dropped.
- **`octo init` / REPL `/init`** (#80) ‚Äî analyses the codebase (entry points, build commands, conventions, common gotchas) and writes a per-repo `.octorules` so future sessions in the same directory have the right context up front. CLI form runs as a one-shot agent loop; REPL `/init` does the same from inside a live session.
- **C8 ‚Äî cross-session user memory** (#79) ‚Äî global `~/.octo/octorules.md` is layered into the system prompt below the project's `.octorules` so personal conventions follow the user across repos. Both files support `@include path` directives (with cycle / depth guards) so a tree of rule files can be composed without duplication.
- **C10 ‚Äî graceful Ctrl-C interrupt** (#77) ‚Äî SIGINT during a turn cancels just that turn (the loop catches `context.Canceled`, finalizes well-formed history with an `[Interrupted by user.]` note, and returns to the prompt). SIGINT at an idle prompt exits cleanly. History stays well-formed across interrupts ‚Äî no orphan `tool_use` without a matching `tool_result`.
- **C12 ‚Äî background command execution + `terminal_output`** (#76) ‚Äî `terminal` gains `background: true` mode that decouples a long-running command from the 30s synchronous timeout; returns a background process id. New `terminal_output(process_id)` tool reads accumulated stdout/stderr + status (analogous to Claude Code's `BashOutput`). Background processes are killed on session exit so nothing outlives the REPL.
- **Anthropic extended thinking** (#74) ‚Äî round-trips thinking blocks across tool calls (preserves `signature` + content per Anthropic's caching requirement so the thinking can be cited in a follow-up). REPL dims the thinking text with a `üßÝ` marker so it's visually distinct from regular output. `--thinking-budget` flag controls how many thinking tokens the model gets per turn.
- **Cache-hit observability** ‚Äî both providers now parse the cache-accounting fields the backends report (Anthropic `cache_creation_input_tokens` / `cache_read_input_tokens`; OpenAI/DeepSeek `prompt_cache_hit_tokens` / `prompt_tokens_details.cached_tokens`) into `Response`/`Reply`. The REPL prints a per-turn `‚ìò cache: N read, M write` line when caching is active, `/cost` shows cumulative cache read/write, and single-turn mode prints a `[usage]` line to stderr. Makes the A2/C4/C2 caching work measurable (and is the prerequisite for tuning the C6 default-on compaction threshold).
- **Environment context in the system prompt** (parity C1) ‚Äî the composed prompt now includes a snapshot of the working environment: cwd, git branch + clean/dirty state, today's date, and OS/arch. Previously the model had no idea where it was running. Taken once at session start (the prompt is frozen per session), so it stays cache-stable across turns. Both Claude Code and Codex inject equivalent context.
- **OpenAI `prompt_cache_key`** (parity C2) ‚Äî chat requests now carry a stable per-conversation `prompt_cache_key` (forwarded via `provider.Request.CacheKey`), so OpenAI routes every turn and tool-loop iteration of a session to the same prompt cache for a higher hit rate. Anthropic ignores it (it uses inline `cache_control`). Matches Codex's session-keyed approach.
- **Micro-compact (per-tool-result truncation)** (parity C5) ‚Äî a single oversized tool result (multi-MB file read, thousands of grep hits, chatty build log) is now truncated middle-out to a `ToolResultMaxBytes` (40 KB) backstop before it enters history, so one pathological call can't dominate the context window. Truncation happens at production time (in `dispatchTools`), not by rewriting old history, so it never mutates already-sent messages and thus never invalidates the C4 conversation cache; the user still sees the full output live via streaming progress. Head and tail are kept (errors/exit codes usually land at the end) and trimmed to valid UTF-8 rune boundaries. Independent of full history compaction.
- **History compaction** ‚Äî opt-in mechanism to keep long sessions from overflowing the context window (and growing cost quadratically). `octo chat --compact-threshold N` (or `Agent.CompactThreshold`): when a turn's input exceeds N tokens, the next turn first summarizes the older history into a single carry-forward message and keeps the last few turns verbatim. Compaction runs only at a safe between-turns boundary and splits on a plain user turn, so `tool_use`/`tool_result` pairs are never severed; a failed summarization is non-fatal (the turn proceeds with full history). Default is 0 (disabled) pending real-model tuning of the threshold + summary quality. (Insert-then-compress per the Ruby cache article; the cache-reuse optimization for the summarization call is a follow-up.)
- **System prompt composition** ‚Äî new `internal/prompt` package assembles the session system prompt from three layers in order of specificity: an embedded base prompt (octo identity + tool-use / permission / read-before-write norms) < the repo's `.octorules` (if present in cwd) < the `--system` flag. Composed once per session and frozen (recomputing mid-session would bust the provider prompt cache). Previously the default system prompt was empty ‚Äî the agent got tool schemas but zero behavioral guidance.
- **Anthropic prompt caching** ‚Äî requests place `cache_control: ephemeral` breakpoints on (a) the system block ‚Äî which, per the tools ‚Üí system ‚Üí messages prefix order, caches the entire stable system+tools prefix ‚Äî and (b) the **last two conversation messages** (parity C4), so the growing history prefix is cached turn-over-turn, not just the static prefix. Two consecutive message markers keep a cache anchor alive across the old-tail/new-tail boundary as history grows. Falls back to marking the last tool when there's no system prompt. Cuts input-token cost dramatically on multi-tool, multi-turn sessions where re-sent history dominates.
- **Go module scaffold** at `github.com/Leihb/octo-agent` (`cmd/octo` entry, `internal/{agent,provider,tools,version}`, Makefile, Go 1.22 CI matrix on Linux / macOS / Windows).
- **`octo chat` CLI** ‚Äî single-turn and interactive REPL modes. Streams by default; `--stream=false` opts back into buffered output.
- **Anthropic Messages provider** ‚Äî `x-api-key` auth, `anthropic-version` header, `content[].text` block parsing. `ANTHROPIC_BASE_URL` env override targets compatible third parties (DeepSeek, Kimi, OpenRouter Anthropic shim, etc.).
- **OpenAI Chat Completions provider** ‚Äî Bearer auth, `system` carried as `messages[0]`, `choices[0].message.content` parsing. `OPENAI_BASE_URL` env override targets compatible third parties (DeepSeek, Bailian/Qwen, vLLM, etc.).
- **Streaming SSE** ‚Äî native aggregators for both protocols (`content_block_delta`/`message_delta` for Anthropic, `chat.completion.chunk` for OpenAI with `[DONE]` sentinel tolerated). `Provider.SendStream` + agent-level `StreamingSender` / `Agent.TurnStream`.
- **Tool calling (agentic loop)** ‚Äî `Agent.Run` / `Agent.RunStream`, normalized `ContentBlock` (`text` / `tool_use` / `tool_result`), provider-side tool-call decoding (OpenAI `finish_reason:"tool_calls"` normalised to `"tool_use"`).
- **`terminal` tool** ‚Äî first concrete tool; runs `sh -c <command>` with a 30s timeout, returns combined stdout+stderr, surfaces non-zero exits as `[exit: ...]` annotations rather than Go errors so the LLM can read and adapt.
- **Session persistence** ‚Äî JSON sessions under `~/.octo/sessions/<YYYYMMDD-HHMMSS>.json`, resume via `octo chat -c <id>`, list via `--list-sessions`, opt out with `--no-save`.
- **REPL slash commands** ‚Äî `/help`, `/cost` (token + USD estimate, per-model pricing), `/save`, `/sessions`, `/exit`, `/quit`.
- **AgentEvent structured event stream** ‚Äî `Agent.RunStream` takes an `EventHandler` that receives typed events (`text_delta`, `tool_input_delta`, `tool_started`, `tool_progress`, `tool_done`, `tool_error`, `turn_done`). Tool events carry `ToolID` + `ToolName` + `Input`; `Output` is truncated to 512 bytes for UI/IM previews while the agent's conversation history keeps the full result. `tool_input_delta` fires zero-or-more times while the LLM streams in a tool_use block's arguments; `tool_progress` fires zero-or-more times between started and done, carrying incremental output (`Chunk` field) from tools that implement `StreamingToolExecutor`.
- **StreamingToolExecutor interface** ‚Äî optional extension to `ToolExecutor` letting tools emit `tool_progress` events as their output arrives. `dispatchTools` type-asserts and routes to `ExecuteStream` when the executor supports it AND the agent has a handler; otherwise falls back to non-streaming `Execute`.
- **`terminal` tool streams stdout+stderr line-by-line** ‚Äî merges both streams through an `io.Pipe`, forwards each line to the progress callback as it arrives, and returns the full aggregated output. Non-zero exits still surface as `[exit: ...]` annotations in the result, not Go errors.
- **`provider.StreamCallbacks` struct** ‚Äî replaces the single `onChunk func(string)` argument to `StreamingProvider.SendStream`. Fields: `OnText` for text deltas (same as before) plus `OnToolDelta(toolID, toolName, partialJSON string)` for argument-streaming. Future event classes (citations, thinking blocks, image deltas) add fields here without bumping the interface. Both Anthropic (`input_json_delta`) and OpenAI (`tool_calls[].function.arguments` chunks) providers now fire `OnToolDelta`.
- **`agent.ToolInputDeltaFunc` + `ToolStreamingSender.StreamMessagesWithTools` extra param** ‚Äî the agent's sender contract grows a second callback alongside `onChunk`. `providerSender` translates between the agent-level (positional callbacks) and provider-level (StreamCallbacks struct) shapes.
- **Core tool suite** ‚Äî seven new built-in tools available under `octo chat --tools`:
  - `read_file` ‚Äî cat-style line-numbered read with offset/limit, capped at 2000 lines per call. Refuses binary extensions (`.exe` / `.zip` / `.png` / `.pdf` / `.sqlite` / ‚Ä¶) and blocking device files (`/dev/random`, `/dev/tty`, etc) by extension before opening, so the LLM doesn't blow context on a multi-MB binary or hang reading `/dev/urandom`. `/dev/null` is permitted (returns EOF).
  - `write_file` ‚Äî write/overwrite with `mkdir -p` semantics on the parent path.
  - `edit_file` ‚Äî exact substring replacement; requires a unique match unless `replace_all=true`. Windows files with CRLF line endings are editable: matching is done in normalized LF space (an `old_string` copied from `read_file` output works), and the result is written back with CRLF preserved if the original used it.
  - `glob` ‚Äî file pattern matching with `**` support, sorted by mtime descending, skips `.git`/`node_modules`/`vendor`/`.venv`, capped at 200 results.
  - `grep` ‚Äî `ripgrep` wrapper with `content` / `files_with_matches` / `count` modes, supports `-A`/`-B`/`-C` context and include-glob filtering. Sets `--max-columns 500` so a hit on a minified JS bundle or base64 blob doesn't flood the LLM's context with one line.
  - `web_fetch` ‚Äî URL ‚Üí Markdown via the Jina Reader proxy, capped at ~200 KB with a clear truncation marker.
  - `web_search` ‚Äî five-backend chain with priority `BRAVE > TAVILY > SERPER > DuckDuckGo HTML > Bing HTML`. Default is **zero-key**; paid backends opt in via env vars. The response `Provider` field tells the LLM which backend produced the results so it can reason about result quality. Inherits the hard-won HTML-scraping rules (no `Accept-Encoding` on Bing, browser-shaped header set, Bing `ck/a?u=a1<base64>` link decoding, DDG 10-minute cooldown on failure).
- **Tool registry** ‚Äî new `internal/tools/registry.go` houses `DefaultRegistry` and `DefaultTools()` as a slice-driven dispatcher. Adding a new built-in tool means one entry in `allTools`.
- **TUI diff card for `edit_file`** ‚Äî `internal/tui` renders `edit_file` tool events as a Claude-Code-style card with `‚óè Update(path)` header, `‚îî Added N, removed M` summary row, line-numbered diff body with red/green washes for `-`/`+` rows, and Chroma syntax highlighting that survives Lipgloss row backgrounds (we re-emit the bg ANSI after each Chroma reset). Other tools still render as the terse `‚Ü≥` status line. New `--plain` flag on `octo chat` forces the `‚Ü≥` fallback for every tool. Companion dev-only command `cmd/tui-preview` (`go run ./cmd/tui-preview`) renders sample cards so visual regressions can be eyeballed locally.
- **REPL renders `tool_progress` and `tool_input_delta`** ‚Äî progress chunks from streaming tools (currently `terminal`) print under their `‚Ü≥` status row with a `‚îÇ` prefix, so a long shell command's output appears line-by-line instead of in a single dump after `tool_done`. While the LLM streams a tool's input arguments, the handler emits a `‚ãØ ¬∑¬∑¬∑¬∑¬∑` typing indicator (capped at 40 dots) that closes when `tool_started` fires. Both behaviours are suppressed for card-rendering tools (e.g. `edit_file`) so the card stays the single visual artifact for those calls.
- **`internal/permission` engine** ‚Äî first slice of M6.5 (tool-safety + permission layer). YAML rule engine evaluates each `tool name √ó input` against an ordered list of `allow | deny | ask` rules; first match wins, no match = implicit ask. `~/.octo/permissions.yml` overrides the embedded `defaults.yml` per-tool. Default rules deny absolute disasters (`rm -rf /`, fork bombs, writes to `~/.ssh/**` / `/etc/**` / `**/.env*`, fetches to RFC1918 / `169.254.169.254` cloud-metadata), allow common safe commands (`ls`, `git status`, `go test`), and ask for everything else risky (`rm -rf`, `git push --force`, `sudo`, `curl`, outside-CWD writes). Two modes: `interactive` (CLI default ‚Äî ask ‚Üí prompt) and `strict` (M8 / M9 default ‚Äî ask ‚Üí deny + structured error so the LLM can explain to its upstream caller). Session-scoped `Remember` cache for "always allow this turn / session" answers.
- **Permission gating wired into the agent loop** ‚Äî second slice of M6.5. `agent.PermissionGate` interface; `Agent.Gate` (nil = no gating, pre-M6.5 behaviour) is consulted in `dispatchTools` immediately before every tool call. A denied call never reaches the executor ‚Äî the loop synthesises an `IsError` tool_result carrying the reason, so the LLM sees the denial and can adapt rather than the run aborting. The CLI builds a `cliPermissionGate` over the engine: `allow` passes through, `deny` returns the structured reason, `ask` (interactive only) prompts `[y]es / [a]lways this session / [N]o` on the shared REPL stdin (answering "always" caches the decision via `Remember`). New `octo chat --permission-mode {interactive|strict}` flag; an invalid value fails closed (exit 2) rather than silently dropping to the more-permissive mode. Gating is active whenever `--tools` is on.
- **Read-before-write enforcement** ‚Äî third slice of M6.5. New `tools.ReadTracker` (one per session, held by `tools.NewDefaultRegistry()`) records every successful `read_file` / `write_file` / `edit_file` with the file's mtime. `write_file` / `edit_file` to an *existing* file is refused unless the file was read this session (`"File has not been read yet‚Ä¶"`) or if its mtime advanced out-of-band since the read (`"File has been modified since‚Ä¶"`) ‚Äî mirroring Claude Code's wording so the LLM re-reads then retries. Creating a brand-new file needs no prior read; a write/edit re-stamps the tracker so a write‚Üíedit sequence doesn't demand a redundant read. The zero-value `DefaultRegistry{}` keeps the old no-enforcement behaviour for tests.
- **Terminal danger detection** ‚Äî fourth slice of M6.5. (1) Credential-path access via `terminal` now asks even when the command verb is otherwise auto-allowed ‚Äî `/etc/passwd`, `/etc/shadow`, `.ssh/id_`, `.aws/credentials`, `id_rsa`, `id_ed25519` patterns sit ahead of the safe-verb allows in `defaults.yml`, closing the hole where `allow: cat ` would let `cat /etc/passwd` slip through. (2) In-place stream edits (`sed -i`, `sed -i.bak`, `sed --in-place`, bundled `-ni`) are denied in `defaults.yml` and additionally guarded inside the terminal tool with a message steering the model to `edit_file` ‚Äî so the change goes through permission + diff + read-before-write rather than bypassing them. The guard stops at shell separators (`| ; &`) so a non-in-place pipeline doesn't false-positive.
- **Tool input hardening** ‚Äî final slice of M6.5, three independent guards. (1) `write_file` refuses content matching high-confidence secret shapes (PEM private-key blocks, GitHub `ghp_`/`github_pat_` tokens, `aws_secret_access_key` assignments, Slack `xox*` tokens); bare AWS access-key IDs like the docs `AKIAIOSFODNN7EXAMPLE` are deliberately *not* flagged to avoid false positives on tutorials. (2) `web_fetch` uses a dedicated HTTP client that refuses to follow a **cross-host** 3xx redirect (same-host path redirects still followed) ‚Äî closes the SSRF/exfil vector where the proxy bounces the fetch to an unexpected host; `web_search` keeps the plain client since search backends legitimately redirect across hosts. (3) `read_file` rejects UNC / network paths (`\\server\share`, `//server/share`) before resolving, since on Windows they trigger an SMB connection that can leak NTLM credentials.
