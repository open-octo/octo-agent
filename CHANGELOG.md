## [Unreleased — 0.5.0-dev]

### Added
- **TUI live background-process panel.** A `background (N running)` panel now appears in the TUI while detached `terminal background:true` commands are in flight — each line shows a spinner, the command, and elapsed time, refreshed by the same tick loop (which now keeps animating between turns while any background process is alive). Completion still drops a scrollback notice (and injects the result into the conversation); the panel just makes the *running* set visible, which the prior fire-and-forget flow couldn't show. New `BackgroundManager.ListRunning()` / package-level `tools.RunningBackground()`. Follow-up to the terminal-notify work + the TUI UX upgrade.
- **TUI panels + theme (TUI UX upgrade, P5 — final).** The queue and the permission / question modals now render as rounded-border panels (the queue gets a titled `queue (N)` panel; modals get a bordered box) instead of loose text. A shared `internal/tui.Panel` / `Box` helper and an adaptive `ColBorder` (light/dark) centralise the chrome, and the input box border now references it too. The true-colour diff/output cards stay github-dark-tuned for now (re-theming those to adaptive is noted as future work). Completes the phased TUI UX upgrade (P1 cards → P2 status bar + input box → P3 spinner → P4 markdown → P5 panels) from `dev-docs/tui-ux-upgrade-design.md`.
- **TUI markdown rendering (TUI UX upgrade, P4).** Assistant replies in the interactive TUI are now rendered as markdown via `glamour` — headings, bold/italic, lists, blockquotes, and syntax-highlighted code blocks. To keep streaming smooth without rendering half a block, text accumulates and is committed one block at a time at the last blank line outside a code fence (the in-progress block stays raw in the live region until it closes); the final block flushes at turn end, and the pre-tool block flushes when a tool starts. Best-effort: any glamour error falls back to the raw text so a turn never breaks over formatting. `--plain` keeps the previous raw line-by-line output. New dependency: `github.com/charmbracelet/glamour` (shares chroma with the diff cards).
- **TUI animated spinner + live tool indicator (TUI UX upgrade, P3).** A `tea.Tick` animation loop (only while a turn is in flight) now drives a braille spinner and a continuously-updating elapsed clock: the initial wait shows `⠋ Thinking (3s)` with a slowly-rotating phrase, the status bar's elapsed time ticks smoothly, and a running card tool (which suppresses its started line) now shows a live `⠙ Run(go test ./...) (4s)` indicator until its finished card commits — closing the P1 gap where long card tools showed nothing until completion. Reuses the plain spinner's braille frames for a consistent look.
- **TUI status bar + bordered input box (TUI UX upgrade, P2).** The interactive TUI's input line is now a rounded-border box sized to the terminal width, with a status bar beneath it showing the model, home-abbreviated working directory, context-window usage (`ctx N%`), estimated session cost (`$…`), permission mode, and per-turn elapsed time; the contextual key hint moved onto a dim line below. New `Agent.ContextUsage()` exposes the context gauge (last-sent tokens / model window). Borderless fallback before the first window-size message or on a very narrow terminal.
- **TUI tool cards (TUI UX upgrade, P1).** The interactive bubbletea TUI now renders tool activity as rich cards instead of one-line `↳ tool ✓` status: `edit_file` shows a chroma-highlighted diff card (the `internal/tui` renderer that previously only ran in the non-TTY fallback view), and `terminal` / `grep` / `glob` / `read_file` / `web_fetch` / `web_search` show an output-preview card (`● Verb(target)` header above the output behind a `│` gutter, capped with a "N more lines" marker). A new `internal/tui` theme skeleton centralises the palette. First of the phased TUI upgrade in `dev-docs/tui-ux-upgrade-design.md` (P2 status bar + bordered input, P3 rich spinner, P4 glamour markdown, P5 panels follow).

### Changed
- **Cards are TUI-only; the plain / non-TTY path is always one-line tool status.** Rich cards are ANSI true-colour boxes — noise in pipes / CI / mswe / redirected output — so `replToolEventHandler` (the `plainView` / `--no-tui` path) now renders every tool, including `edit_file`, as a terse `↳` line; the diff card moved into the TUI. `--plain` now means "one-liners everywhere" and disables the TUI cards too (`cfg.plain`). Headless/piped output is cleaner and more parseable as a result.
- **Built-in tools are on by default** — `octo chat`'s interactive REPL is now a full agent out of the box: the built-in tools (terminal, edit_file, web_search, …), every configured MCP server, and skill execution are all enabled without a flag. This drops the biggest first-run friction for new users — a bare `octo chat` is the complete experience rather than a plain chat box that needs `--tools` to do anything. Risky tool calls still gate through the permission engine (default `interactive`, which prompts on ask), so the safety posture is unchanged. The opt-out is the new `--no-tools` flag (mirrors `--no-memory`), which disables tools, MCP, and skill execution for a plain-chat session. Single-turn mode (`octo chat "message"`) stays tool-free as before — tools run in the REPL loop only. This supersedes the 0.4.0 "resume auto-enables `--tools`" footgun fix: with tools on by default the garbled-XML failure can't occur on the common path, so the auto-enable was removed; resuming a tool-using session with an explicit `--no-tools` now prints a one-line stderr warning instead of silently overriding the choice.

### Added
- **Background-process completion notifications.** A `terminal background:true` command finishing used to be invisible until the model happened to poll `terminal_output` — so "fire off N background jobs and react as each finishes" didn't really work. The `BackgroundManager` now fires an `onExit` hook when a detached process exits; the REPL wires it to (1) inject a `<system-reminder>` notice into the conversation via the existing steer path — folded into the next tool-batch boundary if a turn is running, or prepended to the next turn when idle — so the model is told the command finished, its exit status, and any output since the last read; and (2) in the TUI, print a one-line scrollback notice. `terminal_output` still works for mid-run polling and is deduped against the push (the readNew cursor advances either way). This makes multi-background-process concurrency usable: launch several detached commands and let each completion come back as a notice. Designed in `dev-docs/terminal-notify-parallel-design.md`; builds on the queue/steer/interrupt primitives from `tui-input-modes`.
- **`octo config` — persisted CLI defaults (`~/.octo/config.json`).** First-run friction was "export the API key and pass `--provider`/`--model` every session." `octo config` now runs an interactive wizard that saves your default provider, model, and optional base URL to `~/.octo/config.json` (mode 0600); `octo config show` prints the effective settings and where each comes from (never the key itself), and `octo config path` prints the location. Resolution precedence across `chat` / `init` / `task` is now **CLI flag > env var > config file > built-in default** via a single shared `resolveProviderModel` helper; a stored model/base-URL is scoped to its provider so `--provider <other>` can't carry one vendor's model or endpoint onto another. API keys still read from `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` first — the wizard can optionally store one in the file but defaults to declining (plaintext on disk is discouraged). New `internal/config` package; `--provider` defaults changed from `"anthropic"` to `""` (unset) so the config fallback can take effect, with the resolved default unchanged.
- **First-run orientation in the REPL.** A genuinely new user (no saved sessions yet) now sees a two-line hint on their first `octo chat` — that tools are on, that risky actions ask for approval, and that `octo config` sets defaults. Suppressed for everyone with prior sessions so it never nags, and only shown when tools are enabled. The stale top-level tagline "(Go rewrite in progress)" is gone.
- **Turn-boundary memory nudge** — appended to each user message (wrapped in a `<system-reminder>` block) when cross-session memory + tools are both on. The nudge reminds the model to scan the message it just received for the four signals base.md already documents (preference / correction with WHY / validated approach / external resource) and call `remember` if any matches. Without this, the in-session `remember` tool was being skipped almost universally — base.md's instructions are near the top of a long system prompt, far from the actual decision point, and the default action (do nothing) is free. Concrete case the nudge addresses: user pushes back with "不用，这个不算 bug，确实承诺只应该考虑 agent 给出的" (a domain rule worth saving as feedback), the agent acknowledges and moves on, the rule is forgotten next session. The nudge gates on `len(cfg.tools) > 0` because reminding the model to call a tool it doesn't have is pointless.

### Fixed
- **Streaming responses now have a read/idle timeout — a stalled model stream no longer hangs octo forever (#146).** The streaming HTTP client deliberately carries no end-to-end `Timeout` (streams are long-lived), and the SSE reader did a plain blocking `read()` with no per-read deadline — so a server that accepted the connection and then went silent without closing it left `scanner.Scan()` blocked indefinitely (observed as an 11-minute, ~0-CPU hang on an idle ESTABLISHED socket during an unattended eval batch). Both providers now wrap the response body in a new `retry.IdleTimeoutReader`: each read is raced against a per-read timer that resets on every received chunk, and if no bytes arrive within the idle window (`DefaultStreamIdleTimeout`, 120s; per-client override via `StreamIdleTimeout`, negative disables) the request context is cancelled and the turn fails cleanly with `retry.ErrStreamIdle`. This is distinct from the #138 request-establishment retry, which doesn't cover a mid-stream stall. A mid-stream stall is intentionally **not** auto-retried — the turn has already emitted tokens, so a retry would duplicate them; failing cleanly lets the agent loop roll back and an unattended caller decide whether to re-run. The eval harness's `--octo-timeout` is now a backstop rather than the only guard.

## [0.4.0] — 2026-05-28

**MCP client release.** Adds a full Model Context Protocol client — stdio + Streamable HTTP transports, tools/resources/prompts surfaces, and Device Authorization Grant OAuth so spec-compliant remote servers (Lark Project's MCP server is the smoke-test target) work end-to-end without manual token wrangling. Disk format and existing CLI surface stay unchanged from 0.3.0 — drop a `~/.octo/mcp.json` in and you're set.

### Fixed
- **Resume without `--tools` no longer produces garbled XML tool calls.** When a tool-enabled session was resumed without re-passing `--tools`, the model saw prior `tool_use` blocks in history but no tools array in the new request, and fell back to emitting tool calls as text (a wall of `<tool_calls><invoke name="terminal"><parameter name="command">…`). `octo chat -c <id>` now pre-loads the session, checks for any `tool_use` block, and auto-enables `--tools` with a one-line notice. Plain (no-tools) sessions resume unchanged.
- **First character of the assistant's reply no longer gets eaten by the spinner.** The "thinking…" spinner's `Stop()` was asynchronous — it closed the stop channel and returned, leaving the loop goroutine to race the caller's `Fprint`. The visible failure: the spinner painted a frame, the first text delta arrived and got written right after it, then the loop's `\r\x1b[K` cleared the whole line and ate the leading "Let " (or "I", or whatever the model opened with). `Stop()` now waits for the loop goroutine to actually exit before returning, so the order is always (clear) → (Fprint). Synchronous, idempotent, only the first `Stop()` blocks.

### Added
- **MCP OAuth (Device Authorization Grant)** — HTTP MCP servers that respond `401 Bearer resource_metadata=…` (the RFC 9728 + 8414 discovery dance the MCP spec standardised on) now work end-to-end. On `auth: "oauth"`, the client auto-discovers the authorization server metadata, runs RFC 7591 dynamic client registration, runs RFC 8628 Device Authorization Grant (open URL + enter code — the right UX for a CLI, no local listener), caches the access + refresh tokens at `~/.octo/mcp-tokens/<server>.json` (mode 0600), and silently refreshes on expiry. The 401-retry path means a real expiry mid-session also recovers. Smoke-tested against `https://project.larksuite.com/mcp_server/v1` (Lark Project's MCP server). Tolerant of the common-in-the-wild quirk where servers return HTTP 200 with `{"error":"authorization_pending"}` instead of the spec-required 4xx — the failure mode without that tolerance is "silent empty-token cache, every subsequent call 401s". New config field: `"auth": "oauth"` on an HTTP server entry; default unchanged. Not implemented: Authorization Code + PKCE (would need a local listener; Device Code is equivalent UX without it).
- **MCP client** — `octo chat --tools` now auto-connects to every Model Context Protocol server listed in `~/.octo/mcp.json` (user-global) and `./.octo/mcp.json` (project-local, overrides per name); both stdio (subprocess) and Streamable HTTP transports are supported per MCP spec 2024-11-05. The new `internal/mcp` package is a self-contained client: JSON-RPC 2.0 framing, transport interface with stdio + HTTP implementations, lifecycle handshake, and typed wrappers for tools/list, tools/call, resources/list, resources/read, prompts/list, prompts/get. Each connected server's surface rides alongside octo's built-in tools as `mcp__<server>__<tool>` (matching Claude Code's naming convention so user tooling that grep's tool names keeps working); resources become a synthesized `mcp__<server>__resource_read(uri)` tool and prompts a `mcp__<server>__prompt_get(name, arguments)` tool, gated on the server's advertised capabilities. Config format mirrors Claude Code's `mcpServers` shape (`{command, args, env}` for stdio, `{url, headers}` for HTTP, plus `disabled: true`) so users can copy-paste configs between the two tools. Connect failures degrade — a misconfigured or slow server is logged on stderr and skipped (10s per-server timeout); the session keeps going with the survivors. New `/mcp` REPL slash command lists connected servers + their surface counts + the server-supplied instructions. `octo help mcp` documents the config format, transports, and the v1 omissions (notifications, subscriptions, sampling, roots).

## [0.3.0] — 2026-05-28

CLI usability sweep — five PRs improving the `octo chat` / `octo task` surface without touching the underlying agent loop. The headline change is that interactive sessions feel like a modern CLI: readline-edited prompt with history, 8-character short IDs everywhere a 23-character ID used to be, `octo help <command>` pages with copy-paste examples, a `thinking…` spinner during pauses, `--quiet` / `--verbose` knobs, and TAB completion for bash / zsh / fish that knows your sessions and tasks.

### Added
- **Shell completion for bash / zsh / fish (UX-5)** — new `octo completion bash|zsh|fish` prints a tiny per-shell snippet the user sources from their rc file (`source <(octo completion bash)`, or persist with `octo completion zsh > "${fpath[1]}/_octo"`, or `octo completion fish > ~/.config/fish/completions/octo.fish`). The snippet delegates each TAB to a hidden `octo __complete <command-line-so-far>` subcommand so all routing lives in one Go function — three shell scripts stay near-identical and don't drift as new flags land. Static completion covers top-level subcommands, `task` / `memory` / `memoryd` / `help` / `completion` subcommands, `--provider` (anthropic|openai), and `--permission-mode` (interactive|strict). **Dynamic completion** reads the live filesystem: TAB after `octo chat -c ` lists `last` + the short + full IDs of the 50 most recent sessions; TAB after `octo task status|show|resume|cancel|run ` does the same for tasks. `octo help completion` documents installation per shell.
- **Spinner + `--quiet` / `--verbose` (UX-4)** — `octo chat` now shows a Braille spinner with a "thinking…" label during the pause between turn start and the first byte of model output. Auto-disables itself when stdout isn't a tty (pipes, tests, redirected output stay free of escape codes), and never appears for sub-250ms responses (250ms grace period before the first frame). Two new flags + one env var control the chrome level: `--quiet` (or `OCTO_VERBOSITY=quiet`) strips the spinner, startup banner, per-turn cache line, single-turn `[usage]` line, and the "Session saved →" footer — useful for clean piped output and scripted runs; `--verbose` (or `OCTO_VERBOSITY=verbose`) adds an expanded startup block (model, permission mode, tool count) and turns the per-turn cache line into an always-on debug signal. CLI flags beat env; `--quiet` beats `--verbose` when both are set. Defaults are unchanged — existing users see only the new spinner during pauses.
- **`octo help <command>` + actionable errors (UX-3)** — new `octo help chat`, `octo help task`, `octo help memory`, `octo help init`, `octo help memoryd` print a rich per-command page with copy-paste examples, the most-reached-for flags grouped by purpose, the relevant env vars, and a pointer back to `<command> --help` for the full flag dump. The top-level `octo help` now advertises the per-command form. Three high-pain error sites also gained discovery hints: the missing-API-key first-run path on both providers now shows the signup URL plus the alternative-provider workaround (`octo: ANTHROPIC_API_KEY is not set. … export ANTHROPIC_API_KEY=sk-ant-… or octo chat --provider openai`); `octo chat -c <bad-id>` tells the user to run `--list-sessions`; every `octo task <verb> <bad-id>` site (status / show / resume / cancel / run) tells the user to run `octo task list`. No flag surface changed — only stderr content and the new `help` subcommand routing.
- **Short IDs + `last` shortcut everywhere (UX-2)** — `octo sessions list`, REPL `/sessions`, and `octo task list` now show the 8-character hex suffix of each ID as the leftmost column (with a human-readable created-at next to it), so the user copies 8 characters instead of 23. Every CLI command that consumes an ID — `octo chat -c <id>`, `octo task status/show/resume/cancel/run` — accepts: the full ID (unchanged), the 8-char short ID, any unique substring of an ID, or the literal `last` to mean "most recent". Ambiguity is an error that lists candidates so the user can re-disambiguate. The disk format is untouched — files are still `YYYYMMDD-HHMMSS-<hex>.{jsonl,json}`, which keeps lexicographic-sort = chronological-sort and means zero migration for existing sessions/tasks. Same idea git uses: long SHA on disk, 7-char short SHA in `git log`.
- **REPL line editing + history (UX-1)** — `octo chat` now drives the prompt through `chzyer/readline` when stdin is an interactive terminal: up/down arrow recall a persistent history at `~/.octo/history` (override via `OCTO_HISTORY_FILE`), Ctrl-A / Ctrl-E / Ctrl-W and the usual readline editing keys work, and a trailing `\` continues the input onto a `... ` continuation line (bash-style; doubled `\\` is a literal). At an idle prompt Ctrl-C clears the line instead of exiting (use `/exit` or Ctrl-D), matching `bash` / `zsh` convention. The asker (`ask_user_question`) and the permission gate share the same line reader, so the `Other (free text)` and `allow? [y/a/N]` prompts get history + editing too. Non-tty stdin (pipes, tests, `octo chat < input.txt`) keeps the existing scanner path verbatim — no behaviour change there. First of five planned CLI-UX PRs; see `dev-docs/go-rewrite-roadmap.md` for the rest (ID shortcuts, error messages + `octo help`, status feedback + verbosity, shell completion).

## [0.2.0] — 2026-05-28

Adds **M11** (autonomous task orchestration) and closes **C9 Phase 2 + Phase 3** (the memory daemon and the turn-boundary hook surface for retrieval layers like Hindsight).

### Added
- **C9 Phase 2 — memory daemon** (#115, #116, #117) — `octo memoryd start|stop|status` runs a long-lived foreground process that takes over the boundary extraction + consolidation passes the chat path runs at startup. Sessions are scanned each tick (15s default); a session whose mtime hasn't moved in 15 minutes is eligible for extraction, so the daemon never races a still-active chat. One session per tick by design — `SIGTERM` never finds the daemon mid-extract for more than one session's worth of LLM time. Provider config read from env (`ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `OCTO_MEMORYD_PROVIDER` override); refuses to start without a key. Chat side detects the daemon via the PID file and skips the in-process memory pass, so REPL startup is effectively instant when the daemon is alive. Windows fail-soft (daemon refuses; chat keeps its Phase 1 path).
- **C9 Phase 3 — turn-boundary hooks** (#118, #119) — `internal/hooks` exposes pre-turn / post-turn shell hooks at the REPL turn boundary, the same surface Claude Code's `UserPromptSubmit` hook uses. The pre-turn hook receives `{"user_input": "..."}` on stdin and its stdout (text or `{"additional_context": "..."}`) is folded into the user message via the canonical divider. The post-turn hook receives `{user_input, assistant_reply}` for retention; output is ignored. Config via env vars (`OCTO_HOOK_PRE_TURN`, `OCTO_HOOK_POST_TURN`, `OCTO_HOOK_TIMEOUT`); zero env vars → zero-cost no-op. 5s default timeout, 30s ceiling, `cmd.WaitDelay = 1s` so a child `sleep` can't keep the pipes open after `SIGKILL`. Pre-turn failures log but never block the turn; post-turn runs only on a successful turn so failed turns don't pollute the retention index. `dev-docs/c9-phase3-hindsight.md` walks through wiring Hindsight (and other retrieval layers) into this surface with copy-paste recall/retain scripts.
- **M11 — autonomous task orchestration** (#110, #111, #112, #113) — `octo task start "<goal>"` decomposes the goal into a DAG of subtasks via a planner side-call, then dispatches one M10 sub-agent per ready node in parallel, fsync'ing state per transition so a `kill -9` resumes cleanly. The harness drives task-graph state — never sends the accumulated conversation to an evaluator (that's the failure mode of `/goal`-style designs on long tasks). Subcommands:
  - `octo task start "<goal>" [--plan-only]` — plan + run end-to-end (or just plan).
  - `octo task run <id>` — execute a previously planned task.
  - `octo task list` (alias `ls`) — newest-first table of every task.
  - `octo task status <id>` — full DAG state with per-subtask glyphs (▶ ○ ✓ ✗ ⊘), `blocked_by` lines, one-line result/error snippets.
  - `octo task show <id> <subtask-id>` — full Description / Started / Finished / Duration / Error / Result for one subtask.
  - `octo task resume <id>` — resets Failed / Skipped subtasks to Pending and re-runs (Done subtasks keep their results).
  - `octo task cancel <id>` — marks a task Cancelled.

  New `internal/taskgraph` package owns the Task / Subtask data model + JSON-with-fsync persistence under `~/.octo/tasks/<id>.json`. `agent.PlanTask` is a new side-call with an 80-line planner system prompt (no-op gate, DAG anti-patterns, evidence→implication shape borrowed from the stage_one memory extract). `taskgraph.Scheduler` walks the DAG with **stop-on-first-failure** semantics: a failed subtask marks the whole task Failed and stamps remaining Pending subtasks as Skipped, so `resume` can distinguish "didn't try" from "tried and failed". Foreground execution only in v1 (no daemon). See `dev-docs/go-rewrite-roadmap.md` §M11 for design + acceptance criteria.

## [0.1.0] — 2026-05-28

First numbered Go release. Contains the entire `octo-agent` rewrite from the Ruby baseline plus 14 milestones of feature work (M2–M7, M10, parity items C1–C12 + C8/C9 typed memory, OS-level sandbox, identity files, ask_user_question, task_manager).

### Fixed
- **SSE parsing tolerant of `data:` without trailing space** (#72) — strict `data: ` decoders missed events from some third-party servers that emit `data:<value>` (no space).
- **Delta-message cache usage parsing** (#72) — Anthropic streams cache token counts in a final `message_delta` event; previously we only read those from `message_start`, so streaming sessions reported zero cache hits.
- **Session ID collisions** — session IDs were `YYYYMMDD-HHMMSS` (second precision), so two sessions started in the same second silently overwrote each other's save file. IDs now carry an 8-hex-char (32-bit) random suffix (`YYYYMMDD-HHMMSS-xxxxxxxx`), keeping them sortable/readable while making same-second collisions negligible.

### Changed
- **History compaction default-on with window-relative threshold** (#75, parity C6) — `--compact-threshold 0` (the default) now means "auto-trigger near `windowSize × 0.8`" computed from the model's context window, instead of "disabled". Negative values still disable; explicit positives override.
- **OpenAI reasoning_content round-trip** (#73) — third-party reasoning-capable models (DeepSeek-R1, Qwen thinking models) emit `reasoning_content` alongside `content`. Both are captured and replayed back in subsequent turns so the model can refer to its own prior reasoning across tool calls.
- **Append-only JSONL session persistence** (parity C7) — sessions are now stored as a JSONL transcript (`~/.octo/sessions/<id>.jsonl`: a meta line + one line per message) instead of a single JSON object rewritten in full on every turn. Each save **appends** only the messages added since the last save (per-turn cost O(new messages), not O(total history)), falling back to a full rewrite only when history shrinks (i.e. compaction). "Last updated" comes from file mtime, so `--list-sessions` no longer needs a stored `updated_at`. **Breaking:** old `.json` session files are not read (no migration); resume-by-id and the `.json`-suffix-tolerant lookup still work for `.jsonl`.
- **Structured compaction summary** (parity C3) — the history-compaction side-call now uses a structured five-section summary prompt (Primary Request / Key Technical Concepts / Files and Code / Errors and Fixes / Current State and Next Steps) instead of a freeform "summarize this" instruction, mirroring Claude Code's compaction prompt — a structured carry-forward retains far more usable signal.
- **Parallel dispatch of read-only tools** — when the model emits several tool calls in one turn and they're all side-effect-free (`read_file` / `glob` / `grep` / `web_fetch` / `web_search`), `dispatchTools` now runs them concurrently instead of one-at-a-time, cutting latency on the common "read three files at once" pattern. The permission gate still runs serially first (an interactive `ask` prompt must not race another on stdin); only the allowed calls execute, and results stay in block order. Any mutating or streaming tool in the batch (or a single call) keeps the serial path, preserving `tool_progress` streaming.
- **Loop budget gates** — the agentic loop's hardcoded 20-iteration cap (which `error`ed out and discarded the turn) is replaced by configurable `Agent.MaxTurns` (default 20) and `Agent.MaxCostUSD` (0 = unlimited), surfaced as `octo chat --max-turns` / `--max-cost`. Hitting either limit now ends the run *gracefully*: history keeps the partial progress and the caller gets a friendly assistant message (`[octo] Stopped: …`) with a sentinel `StopReason` (`max_turns` / `max_cost`) instead of an error. The cost gate checks the running session estimate before each provider call. (Ports Ruby CATCHUP P0-4.)
- **Unified agentic loop** — `Run` (buffered) and `RunStream` (streaming) previously carried two near-duplicate copies of the tool-dispatch loop, so changes like the permission gate had to be applied in both. They now share a single `runLoop`; `Run` drives it with a buffered send closure and a nil event handler (every event-emit path short-circuits on nil), `RunStream` with a streaming closure and a real handler. Behaviour is unchanged — verified by the existing `Run` and `RunStream` test suites.

### Added
- **`task_manager` tool group** (#108) — three tools (`task_create` / `task_update` / `task_list`) let the model break work into discrete trackable steps. Tasks carry an `id` (monotonic, never reused), `subject` (imperative title), `description`, `status` (`pending | in_progress | completed | deleted`), and `active_form` (present-continuous label surfaced when in-progress so the row reads like a spinner). Session-scoped in-memory store; cross-session persistence is M11 territory. `/tasks` REPL slash command renders the same list via the same formatter `task_list` returns to the LLM, so user-facing and model-facing rendering can't drift.
- **`ask_user_question` tool** (#107) — model asks one structured clarifying question with 2-4 options and an automatic "Other (free text)" tail; supports `multi_select` and an optional `header` tag. REPL renders a numbered list, parses comma-separated picks for multi-select, returns `User chose: …` (or `(user cancelled)`) as the tool_result. Gated REPL-only — absent from `DefaultTools()` until an Asker is registered, so single-turn/unattended modes don't advertise a tool that can only error.
- **Sub-agent-backed memory consolidator** (#105) — consolidation now spawns a sub-agent with a read-only allowlist (`read_file` / `grep` / `glob`) that can autonomously navigate the memory dir (grep `rollout_summaries/`, read individual `<slug>.md` files for context) and return the new summary text. Falls back to the existing `ConsolidateMemory` side-call when no Spawner is wired or the sub-agent errors. The parent stays in charge of `WriteSummary` + `ArchiveAll` so the v1 marker + git commits happen in one place.
- **M10 — `launch_agent` sub-agent tool** (#104) — model can spawn a child agent that handles a focused sub-task in isolation and reports a final reply. Child shares the parent's `Sender` (one provider connection) and `System` (same harness identity) but runs on a fresh `History`, capped at 12 turns so a runaway child can't burn budget unnoticed. Token usage rolls back into the parent's session totals via `Agent.AccrueChildUsage` so `/cost` reports one consolidated number. Multiple `launch_agent` calls in one `tool_use` batch fan out concurrently. Recursion guard: `launch_agent` is filtered out of the child's tool list AND a context marker is stamped so a hallucinating child can't recurse.
- **C9 — typed cross-session memory** (#93–#102) — `internal/memory` package owns `~/.octo/memory/`. **Typed entries**: one fact per `<slug>.md` with YAML frontmatter (`name / description / type / created / last_verified`); `type` is `user | feedback | project | reference`. **`remember` tool** writes immediately when the model recognizes a load-bearing signal (preference, correction, project decision, external resource pointer). **Boundary extraction** runs as a side-call at REPL start over the previous session's transcript to sweep what the model didn't record live. **Incremental consolidate** folds new entries into a maintained `memory_summary.md` instead of rebuilding; `ArchiveAll` drops folded entries from the working tree (#96). **Three-piece per-rollout artifacts** (#102) — each consolidation produces `rollout_slug` + `rollout_summary` + `facts[]`; the rollout summary lives in `rollout_summaries/<timestamp>-<slug>.md` as an on-demand reference that future sessions can grep. **Stage_one-style extract prompt** (#98) — explicit no-op gate, outcome triage (success / partial / fail / uncertain), evidence→implication wording for feedback, user > tool > assistant reading priority. **`memory_summary.md` v1 protocol marker** (#100) — `<!-- octo-memory v1 -->` HTML comment as the first line; stripped on read so the LLM never sees it; HTML-comment so Markdown viewers render it invisibly. **Git baseline replaces `archive/` subdir** (#101) — `~/.octo/memory/` is auto-`git init`ed on first write; every Save / WriteSummary / ArchiveAll commits, so git history is the archive. `Store.WorkspaceDiff(baseSHA)` + `LastConsolidatedSHA` are wired for the sub-agent consolidator. **base.md Memory section** expanded to ~60 lines with citations (`(from memory: …)` inline attribution when a recalled fact is load-bearing), verify-first (grep/read remembered paths before acting), and "user contradicts memory → user wins" (#99). REPL `/memory` + `octo memory list [--archive]` inspect the store; `--no-memory` disables. Design + Codex comparison: [`c9-memory-design.md`](dev-docs/c9-memory-design.md).
- **Identity files** (#92) — `~/.octo/soul.md` (agent persona) and `~/.octo/user.md` (user profile) layered into the system prompt below `base.md` so the agent has a consistent voice and the user's name/role/preferences are background context. Full nine-layer composition: `base → soul → env → skills → memory → user.md → octorules(user) → octorules(project) → --system`. Design: [`identity-files-design.md`](dev-docs/identity-files-design.md).
- **M7 — Skill loader** (#87) — Claude Code-compatible `SKILL.md` format (YAML frontmatter + body). Skills discovered at session start from `~/.octo/skills/<name>/` (user) and `./.octo/skills/<name>/` (project) and rendered into a one-line-per-skill manifest in the system prompt (L1 "what's available"). The new `skill` tool serves full bodies on demand (L2 "give me the instructions"). REPL `/skills` lists them; `/<name>` triggers one directly. `--list-skills` enumerates from the CLI. Symlinks under `~/.claude/skills/` work directly (the format is bit-compatible).
- **C11 — OS-level command sandbox** (#82, #84) — opt-in `--sandbox` wraps every `terminal` invocation in macOS Seatbelt (`sbpl` policy) / Linux Landlock + seccomp (with `bwrap` fallback). Default policy: filesystem writes restricted to cwd, network disabled. `--sandbox-write-root` / `--sandbox-read-root` extend the policy with extra paths; `--sandbox-allow-net` toggles network on. Windows degrades to a clear notice. Stays opt-in by deliberate decision (#89) — `c11-sandbox-design.md` §8 documents why default-on was not pursued, and Phase 2 host allowlist (#85) was evaluated and dropped.
- **`octo init` / REPL `/init`** (#80) — analyses the codebase (entry points, build commands, conventions, common gotchas) and writes a per-repo `.octorules` so future sessions in the same directory have the right context up front. CLI form runs as a one-shot agent loop; REPL `/init` does the same from inside a live session.
- **C8 — cross-session user memory** (#79) — global `~/.octo/octorules.md` is layered into the system prompt below the project's `.octorules` so personal conventions follow the user across repos. Both files support `@include path` directives (with cycle / depth guards) so a tree of rule files can be composed without duplication.
- **C10 — graceful Ctrl-C interrupt** (#77) — SIGINT during a turn cancels just that turn (the loop catches `context.Canceled`, finalizes well-formed history with an `[Interrupted by user.]` note, and returns to the prompt). SIGINT at an idle prompt exits cleanly. History stays well-formed across interrupts — no orphan `tool_use` without a matching `tool_result`.
- **C12 — background command execution + `terminal_output`** (#76) — `terminal` gains `background: true` mode that decouples a long-running command from the 30s synchronous timeout; returns a background process id. New `terminal_output(process_id)` tool reads accumulated stdout/stderr + status (analogous to Claude Code's `BashOutput`). Background processes are killed on session exit so nothing outlives the REPL.
- **Anthropic extended thinking** (#74) — round-trips thinking blocks across tool calls (preserves `signature` + content per Anthropic's caching requirement so the thinking can be cited in a follow-up). REPL dims the thinking text with a `🧠` marker so it's visually distinct from regular output. `--thinking-budget` flag controls how many thinking tokens the model gets per turn.
- **Cache-hit observability** — both providers now parse the cache-accounting fields the backends report (Anthropic `cache_creation_input_tokens` / `cache_read_input_tokens`; OpenAI/DeepSeek `prompt_cache_hit_tokens` / `prompt_tokens_details.cached_tokens`) into `Response`/`Reply`. The REPL prints a per-turn `ⓘ cache: N read, M write` line when caching is active, `/cost` shows cumulative cache read/write, and single-turn mode prints a `[usage]` line to stderr. Makes the A2/C4/C2 caching work measurable (and is the prerequisite for tuning the C6 default-on compaction threshold).
- **Environment context in the system prompt** (parity C1) — the composed prompt now includes a snapshot of the working environment: cwd, git branch + clean/dirty state, today's date, and OS/arch. Previously the model had no idea where it was running. Taken once at session start (the prompt is frozen per session), so it stays cache-stable across turns. Both Claude Code and Codex inject equivalent context.
- **OpenAI `prompt_cache_key`** (parity C2) — chat requests now carry a stable per-conversation `prompt_cache_key` (forwarded via `provider.Request.CacheKey`), so OpenAI routes every turn and tool-loop iteration of a session to the same prompt cache for a higher hit rate. Anthropic ignores it (it uses inline `cache_control`). Matches Codex's session-keyed approach.
- **Micro-compact (per-tool-result truncation)** (parity C5) — a single oversized tool result (multi-MB file read, thousands of grep hits, chatty build log) is now truncated middle-out to a `ToolResultMaxBytes` (40 KB) backstop before it enters history, so one pathological call can't dominate the context window. Truncation happens at production time (in `dispatchTools`), not by rewriting old history, so it never mutates already-sent messages and thus never invalidates the C4 conversation cache; the user still sees the full output live via streaming progress. Head and tail are kept (errors/exit codes usually land at the end) and trimmed to valid UTF-8 rune boundaries. Independent of full history compaction.
- **History compaction** — opt-in mechanism to keep long sessions from overflowing the context window (and growing cost quadratically). `octo chat --compact-threshold N` (or `Agent.CompactThreshold`): when a turn's input exceeds N tokens, the next turn first summarizes the older history into a single carry-forward message and keeps the last few turns verbatim. Compaction runs only at a safe between-turns boundary and splits on a plain user turn, so `tool_use`/`tool_result` pairs are never severed; a failed summarization is non-fatal (the turn proceeds with full history). Default is 0 (disabled) pending real-model tuning of the threshold + summary quality. (Insert-then-compress per the Ruby cache article; the cache-reuse optimization for the summarization call is a follow-up.)
- **System prompt composition** — new `internal/prompt` package assembles the session system prompt from three layers in order of specificity: an embedded base prompt (octo identity + tool-use / permission / read-before-write norms) < the repo's `.octorules` (if present in cwd) < the `--system` flag. Composed once per session and frozen (recomputing mid-session would bust the provider prompt cache). Previously the default system prompt was empty — the agent got tool schemas but zero behavioral guidance.
- **Anthropic prompt caching** — requests place `cache_control: ephemeral` breakpoints on (a) the system block — which, per the tools → system → messages prefix order, caches the entire stable system+tools prefix — and (b) the **last two conversation messages** (parity C4), so the growing history prefix is cached turn-over-turn, not just the static prefix. Two consecutive message markers keep a cache anchor alive across the old-tail/new-tail boundary as history grows. Falls back to marking the last tool when there's no system prompt. Cuts input-token cost dramatically on multi-tool, multi-turn sessions where re-sent history dominates.

- **Core tool suite** — seven new built-in tools available under `octo chat --tools`:
  - `read_file` — cat-style line-numbered read with offset/limit, capped at 2000 lines per call. Refuses binary extensions (`.exe` / `.zip` / `.png` / `.pdf` / `.sqlite` / …) and blocking device files (`/dev/random`, `/dev/tty`, etc) by extension before opening, so the LLM doesn't blow context on a multi-MB binary or hang reading `/dev/urandom`. `/dev/null` is permitted (returns EOF).
  - `write_file` — write/overwrite with `mkdir -p` semantics on the parent path.
  - `edit_file` — exact substring replacement; requires a unique match unless `replace_all=true`. Windows files with CRLF line endings are editable: matching is done in normalized LF space (an `old_string` copied from `read_file` output works), and the result is written back with CRLF preserved if the original used it.
  - `glob` — file pattern matching with `**` support, sorted by mtime descending, skips `.git`/`node_modules`/`vendor`/`.venv`, capped at 200 results.
  - `grep` — `ripgrep` wrapper with `content` / `files_with_matches` / `count` modes, supports `-A`/`-B`/`-C` context and include-glob filtering. Sets `--max-columns 500` so a hit on a minified JS bundle or base64 blob doesn't flood the LLM's context with one line.
  - `web_fetch` — URL → Markdown via the Jina Reader proxy, capped at ~200 KB with a clear truncation marker.
  - `web_search` — five-backend chain with priority `BRAVE > TAVILY > SERPER > DuckDuckGo HTML > Bing HTML`. Default is **zero-key**; paid backends opt in via env vars. The response `Provider` field tells the LLM which backend produced the results so it can reason about result quality. Inherits the hard-won HTML-scraping rules (no `Accept-Encoding` on Bing, browser-shaped header set, Bing `ck/a?u=a1<base64>` link decoding, DDG 10-minute cooldown on failure).
- **Tool registry** — new `internal/tools/registry.go` houses `DefaultRegistry` and `DefaultTools()` as a slice-driven dispatcher. Adding a new built-in tool means one entry in `allTools`.
- **TUI diff card for `edit_file`** — `internal/tui` renders `edit_file` tool events as a Claude-Code-style card with `● Update(path)` header, `└ Added N, removed M` summary row, line-numbered diff body with red/green washes for `-`/`+` rows, and Chroma syntax highlighting that survives Lipgloss row backgrounds (we re-emit the bg ANSI after each Chroma reset). Other tools still render as the terse `↳` status line. New `--plain` flag on `octo chat` forces the `↳` fallback for every tool. Companion dev-only command `cmd/tui-preview` (`go run ./cmd/tui-preview`) renders sample cards so visual regressions can be eyeballed locally.
- **REPL renders `tool_progress` and `tool_input_delta`** — progress chunks from streaming tools (currently `terminal`) print under their `↳` status row with a `│` prefix, so a long shell command's output appears line-by-line instead of in a single dump after `tool_done`. While the LLM streams a tool's input arguments, the handler emits a `⋯ ·····` typing indicator (capped at 40 dots) that closes when `tool_started` fires. Both behaviours are suppressed for card-rendering tools (e.g. `edit_file`) so the card stays the single visual artifact for those calls.
- **`internal/permission` engine** — first slice of M6.5 (tool-safety + permission layer). YAML rule engine evaluates each `tool name × input` against an ordered list of `allow | deny | ask` rules; first match wins, no match = implicit ask. `~/.octo/permissions.yml` overrides the embedded `defaults.yml` per-tool. Default rules deny absolute disasters (`rm -rf /`, fork bombs, writes to `~/.ssh/**` / `/etc/**` / `**/.env*`, fetches to RFC1918 / `169.254.169.254` cloud-metadata), allow common safe commands (`ls`, `git status`, `go test`), and ask for everything else risky (`rm -rf`, `git push --force`, `sudo`, `curl`, outside-CWD writes). Two modes: `interactive` (CLI default — ask → prompt) and `strict` (M8 / M9 default — ask → deny + structured error so the LLM can explain to its upstream caller). Session-scoped `Remember` cache for "always allow this turn / session" answers.
- **Permission gating wired into the agent loop** — second slice of M6.5. `agent.PermissionGate` interface; `Agent.Gate` (nil = no gating, pre-M6.5 behaviour) is consulted in `dispatchTools` immediately before every tool call. A denied call never reaches the executor — the loop synthesises an `IsError` tool_result carrying the reason, so the LLM sees the denial and can adapt rather than the run aborting. The CLI builds a `cliPermissionGate` over the engine: `allow` passes through, `deny` returns the structured reason, `ask` (interactive only) prompts `[y]es / [a]lways this session / [N]o` on the shared REPL stdin (answering "always" caches the decision via `Remember`). New `octo chat --permission-mode {interactive|strict}` flag; an invalid value fails closed (exit 2) rather than silently dropping to the more-permissive mode. Gating is active whenever `--tools` is on.
- **Read-before-write enforcement** — third slice of M6.5. New `tools.ReadTracker` (one per session, held by `tools.NewDefaultRegistry()`) records every successful `read_file` / `write_file` / `edit_file` with the file's mtime. `write_file` / `edit_file` to an *existing* file is refused unless the file was read this session (`"File has not been read yet…"`) or if its mtime advanced out-of-band since the read (`"File has been modified since…"`) — mirroring Claude Code's wording so the LLM re-reads then retries. Creating a brand-new file needs no prior read; a write/edit re-stamps the tracker so a write→edit sequence doesn't demand a redundant read. The zero-value `DefaultRegistry{}` keeps the old no-enforcement behaviour for tests.
- **Terminal danger detection** — fourth slice of M6.5. (1) Credential-path access via `terminal` now asks even when the command verb is otherwise auto-allowed — `/etc/passwd`, `/etc/shadow`, `.ssh/id_`, `.aws/credentials`, `id_rsa`, `id_ed25519` patterns sit ahead of the safe-verb allows in `defaults.yml`, closing the hole where `allow: cat ` would let `cat /etc/passwd` slip through. (2) In-place stream edits (`sed -i`, `sed -i.bak`, `sed --in-place`, bundled `-ni`) are denied in `defaults.yml` and additionally guarded inside the terminal tool with a message steering the model to `edit_file` — so the change goes through permission + diff + read-before-write rather than bypassing them. The guard stops at shell separators (`| ; &`) so a non-in-place pipeline doesn't false-positive.
- **Tool input hardening** — final slice of M6.5, three independent guards. (1) `write_file` refuses content matching high-confidence secret shapes (PEM private-key blocks, GitHub `ghp_`/`github_pat_` tokens, `aws_secret_access_key` assignments, Slack `xox*` tokens); bare AWS access-key IDs like the docs `AKIAIOSFODNN7EXAMPLE` are deliberately *not* flagged to avoid false positives on tutorials. (2) `web_fetch` uses a dedicated HTTP client that refuses to follow a **cross-host** 3xx redirect (same-host path redirects still followed) — closes the SSRF/exfil vector where the proxy bounces the fetch to an unexpected host; `web_search` keeps the plain client since search backends legitimately redirect across hosts. (3) `read_file` rejects UNC / network paths (`\\server\share`, `//server/share`) before resolving, since on Windows they trigger an SMB connection that can leak NTLM credentials.
