# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased — 0.1.0-dev]

### Fixed
- **Session ID collisions** — session IDs were `YYYYMMDD-HHMMSS` (second precision), so two sessions started in the same second silently overwrote each other's save file. IDs now carry an 8-hex-char (32-bit) random suffix (`YYYYMMDD-HHMMSS-xxxxxxxx`), keeping them sortable/readable while making same-second collisions negligible.

### Changed
- **Append-only JSONL session persistence** (parity C7) — sessions are now stored as a JSONL transcript (`~/.octo/sessions/<id>.jsonl`: a meta line + one line per message) instead of a single JSON object rewritten in full on every turn. Each save **appends** only the messages added since the last save (per-turn cost O(new messages), not O(total history)), falling back to a full rewrite only when history shrinks (i.e. compaction). "Last updated" comes from file mtime, so `--list-sessions` no longer needs a stored `updated_at`. **Breaking:** old `.json` session files are not read (no migration); resume-by-id and the `.json`-suffix-tolerant lookup still work for `.jsonl`.
- **Structured compaction summary** (parity C3) — the history-compaction side-call now uses a structured five-section summary prompt (Primary Request / Key Technical Concepts / Files and Code / Errors and Fixes / Current State and Next Steps) instead of a freeform "summarize this" instruction, mirroring Claude Code's compaction prompt — a structured carry-forward retains far more usable signal.
- **Parallel dispatch of read-only tools** — when the model emits several tool calls in one turn and they're all side-effect-free (`read_file` / `glob` / `grep` / `web_fetch` / `web_search`), `dispatchTools` now runs them concurrently instead of one-at-a-time, cutting latency on the common "read three files at once" pattern. The permission gate still runs serially first (an interactive `ask` prompt must not race another on stdin); only the allowed calls execute, and results stay in block order. Any mutating or streaming tool in the batch (or a single call) keeps the serial path, preserving `tool_progress` streaming.
- **Loop budget gates** — the agentic loop's hardcoded 20-iteration cap (which `error`ed out and discarded the turn) is replaced by configurable `Agent.MaxTurns` (default 20) and `Agent.MaxCostUSD` (0 = unlimited), surfaced as `octo chat --max-turns` / `--max-cost`. Hitting either limit now ends the run *gracefully*: history keeps the partial progress and the caller gets a friendly assistant message (`[octo] Stopped: …`) with a sentinel `StopReason` (`max_turns` / `max_cost`) instead of an error. The cost gate checks the running session estimate before each provider call. (Ports Ruby CATCHUP P0-4.)
- **Unified agentic loop** — `Run` (buffered) and `RunStream` (streaming) previously carried two near-duplicate copies of the tool-dispatch loop, so changes like the permission gate had to be applied in both. They now share a single `runLoop`; `Run` drives it with a buffered send closure and a nil event handler (every event-emit path short-circuits on nil), `RunStream` with a streaming closure and a real handler. Behaviour is unchanged — verified by the existing `Run` and `RunStream` test suites.

### Added
- **Cache-hit observability** — both providers now parse the cache-accounting fields the backends report (Anthropic `cache_creation_input_tokens` / `cache_read_input_tokens`; OpenAI/DeepSeek `prompt_cache_hit_tokens` / `prompt_tokens_details.cached_tokens`) into `Response`/`Reply`. The REPL prints a per-turn `ⓘ cache: N read, M write` line when caching is active, `/cost` shows cumulative cache read/write, and single-turn mode prints a `[usage]` line to stderr. Makes the A2/C4/C2 caching work measurable (and is the prerequisite for tuning the C6 default-on compaction threshold).
- **Environment context in the system prompt** (parity C1) — the composed prompt now includes a snapshot of the working environment: cwd, git branch + clean/dirty state, today's date, and OS/arch. Previously the model had no idea where it was running. Taken once at session start (the prompt is frozen per session), so it stays cache-stable across turns. Both Claude Code and Codex inject equivalent context.
- **OpenAI `prompt_cache_key`** (parity C2) — chat requests now carry a stable per-conversation `prompt_cache_key` (forwarded via `provider.Request.CacheKey`), so OpenAI routes every turn and tool-loop iteration of a session to the same prompt cache for a higher hit rate. Anthropic ignores it (it uses inline `cache_control`). Matches Codex's session-keyed approach.
- **Micro-compact (per-tool-result truncation)** (parity C5) — a single oversized tool result (multi-MB file read, thousands of grep hits, chatty build log) is now truncated middle-out to a `ToolResultMaxBytes` (40 KB) backstop before it enters history, so one pathological call can't dominate the context window. Truncation happens at production time (in `dispatchTools`), not by rewriting old history, so it never mutates already-sent messages and thus never invalidates the C4 conversation cache; the user still sees the full output live via streaming progress. Head and tail are kept (errors/exit codes usually land at the end) and trimmed to valid UTF-8 rune boundaries. Independent of full history compaction.
- **History compaction** — opt-in mechanism to keep long sessions from overflowing the context window (and growing cost quadratically). `octo chat --compact-threshold N` (or `Agent.CompactThreshold`): when a turn's input exceeds N tokens, the next turn first summarizes the older history into a single carry-forward message and keeps the last few turns verbatim. Compaction runs only at a safe between-turns boundary and splits on a plain user turn, so `tool_use`/`tool_result` pairs are never severed; a failed summarization is non-fatal (the turn proceeds with full history). Default is 0 (disabled) pending real-model tuning of the threshold + summary quality. (Insert-then-compress per the Ruby cache article; the cache-reuse optimization for the summarization call is a follow-up.)
- **System prompt composition** — new `internal/prompt` package assembles the session system prompt from three layers in order of specificity: an embedded base prompt (octo identity + tool-use / permission / read-before-write norms) < the repo's `.octorules` (if present in cwd) < the `--system` flag. Composed once per session and frozen (recomputing mid-session would bust the provider prompt cache). Previously the default system prompt was empty — the agent got tool schemas but zero behavioral guidance.
- **Anthropic prompt caching** — requests place `cache_control: ephemeral` breakpoints on (a) the system block — which, per the tools → system → messages prefix order, caches the entire stable system+tools prefix — and (b) the **last two conversation messages** (parity C4), so the growing history prefix is cached turn-over-turn, not just the static prefix. Two consecutive message markers keep a cache anchor alive across the old-tail/new-tail boundary as history grows. Falls back to marking the last tool when there's no system prompt. Cuts input-token cost dramatically on multi-tool, multi-turn sessions where re-sent history dominates.
- **Go module scaffold** at `github.com/Leihb/octo-agent` (`cmd/octo` entry, `internal/{agent,provider,tools,version}`, Makefile, Go 1.22 CI matrix on Linux / macOS / Windows).
- **`octo chat` CLI** — single-turn and interactive REPL modes. Streams by default; `--stream=false` opts back into buffered output.
- **Anthropic Messages provider** — `x-api-key` auth, `anthropic-version` header, `content[].text` block parsing. `ANTHROPIC_BASE_URL` env override targets compatible third parties (DeepSeek, Kimi, OpenRouter Anthropic shim, etc.).
- **OpenAI Chat Completions provider** — Bearer auth, `system` carried as `messages[0]`, `choices[0].message.content` parsing. `OPENAI_BASE_URL` env override targets compatible third parties (DeepSeek, Bailian/Qwen, vLLM, etc.).
- **Streaming SSE** — native aggregators for both protocols (`content_block_delta`/`message_delta` for Anthropic, `chat.completion.chunk` for OpenAI with `[DONE]` sentinel tolerated). `Provider.SendStream` + agent-level `StreamingSender` / `Agent.TurnStream`.
- **Tool calling (agentic loop)** — `Agent.Run` / `Agent.RunStream`, normalized `ContentBlock` (`text` / `tool_use` / `tool_result`), provider-side tool-call decoding (OpenAI `finish_reason:"tool_calls"` normalised to `"tool_use"`).
- **`terminal` tool** — first concrete tool; runs `sh -c <command>` with a 30s timeout, returns combined stdout+stderr, surfaces non-zero exits as `[exit: ...]` annotations rather than Go errors so the LLM can read and adapt.
- **Session persistence** — JSON sessions under `~/.octo/sessions/<YYYYMMDD-HHMMSS>.json`, resume via `octo chat -c <id>`, list via `--list-sessions`, opt out with `--no-save`.
- **REPL slash commands** — `/help`, `/cost` (token + USD estimate, per-model pricing), `/save`, `/sessions`, `/exit`, `/quit`.
- **AgentEvent structured event stream** — `Agent.RunStream` takes an `EventHandler` that receives typed events (`text_delta`, `tool_input_delta`, `tool_started`, `tool_progress`, `tool_done`, `tool_error`, `turn_done`). Tool events carry `ToolID` + `ToolName` + `Input`; `Output` is truncated to 512 bytes for UI/IM previews while the agent's conversation history keeps the full result. `tool_input_delta` fires zero-or-more times while the LLM streams in a tool_use block's arguments; `tool_progress` fires zero-or-more times between started and done, carrying incremental output (`Chunk` field) from tools that implement `StreamingToolExecutor`.
- **StreamingToolExecutor interface** — optional extension to `ToolExecutor` letting tools emit `tool_progress` events as their output arrives. `dispatchTools` type-asserts and routes to `ExecuteStream` when the executor supports it AND the agent has a handler; otherwise falls back to non-streaming `Execute`.
- **`terminal` tool streams stdout+stderr line-by-line** — merges both streams through an `io.Pipe`, forwards each line to the progress callback as it arrives, and returns the full aggregated output. Non-zero exits still surface as `[exit: ...]` annotations in the result, not Go errors.
- **`provider.StreamCallbacks` struct** — replaces the single `onChunk func(string)` argument to `StreamingProvider.SendStream`. Fields: `OnText` for text deltas (same as before) plus `OnToolDelta(toolID, toolName, partialJSON string)` for argument-streaming. Future event classes (citations, thinking blocks, image deltas) add fields here without bumping the interface. Both Anthropic (`input_json_delta`) and OpenAI (`tool_calls[].function.arguments` chunks) providers now fire `OnToolDelta`.
- **`agent.ToolInputDeltaFunc` + `ToolStreamingSender.StreamMessagesWithTools` extra param** — the agent's sender contract grows a second callback alongside `onChunk`. `providerSender` translates between the agent-level (positional callbacks) and provider-level (StreamCallbacks struct) shapes.
- **Core tool suite** — seven new built-in tools available under `octo chat --tools`:
  - `read_file` — cat-style line-numbered read with offset/limit, capped at 2000 lines per call. Refuses binary extensions (`.exe` / `.zip` / `.png` / `.pdf` / `.sqlite` / …) and blocking device files (`/dev/random`, `/dev/tty`, etc) by extension before opening, so the LLM doesn't blow context on a multi-MB binary or hang reading `/dev/urandom`. `/dev/null` is permitted (returns EOF).
  - `write_file` — write/overwrite with `mkdir -p` semantics on the parent path.
  - `edit_file` — exact substring replacement; requires a unique match unless `replace_all=true`. Windows files with CRLF line endings are editable: matching is done in normalized LF space (an `old_string` copied from `read_file` output works), and the result is written back with CRLF preserved if the original used it.
  - `glob` — file pattern matching with `**` support, sorted by mtime descending, skips `.git`/`node_modules`/`vendor`/`.venv`, capped at 200 results.
  - `grep` — `ripgrep` wrapper with `content` / `files_with_matches` / `count` modes, supports `-A`/`-B`/`-C` context and include-glob filtering. Sets `--max-columns 500` so a hit on a minified JS bundle or base64 blob doesn't flood the LLM's context with one line.
  - `web_fetch` — URL → Markdown via the Jina Reader proxy, capped at ~200 KB with a clear truncation marker.
  - `web_search` — five-backend chain with priority `BRAVE > TAVILY > SERPER > DuckDuckGo HTML > Bing HTML`. Default is **zero-key**; paid backends opt in via env vars. The response `Provider` field tells the LLM which backend produced the results so it can reason about result quality. Inherits the hard-won HTML-scraping rules (no `Accept-Encoding` on Bing, browser-shaped header set, Bing `ck/a?u=a1<base64>` link decoding, DDG 10-minute cooldown on failure).
- **Tool registry** — new `internal/tools/registry.go` houses `DefaultRegistry` and `DefaultTools()` as a slice-driven dispatcher. Adding a new built-in tool means one entry in `allTools`.
- **TUI diff card for `edit_file`** — `internal/tui` renders `edit_file` tool events as a Claude-Code-style card with `● Update(path)` header, `└ Added N, removed M` summary row, line-numbered diff body with red/green washes for `-`/`+` rows, and Chroma syntax highlighting that survives Lipgloss row backgrounds (we re-emit the bg ANSI after each Chroma reset). Other tools still render as the terse `↳` status line. New `--plain` flag on `octo chat` forces the `↳` fallback for every tool. Companion dev-only command `cmd/tui-preview` (`go run ./cmd/tui-preview`) renders sample cards so visual regressions can be eyeballed locally.
- **REPL renders `tool_progress` and `tool_input_delta`** — progress chunks from streaming tools (currently `terminal`) print under their `↳` status row with a `│` prefix, so a long shell command's output appears line-by-line instead of in a single dump after `tool_done`. While the LLM streams a tool's input arguments, the handler emits a `⋯ ·····` typing indicator (capped at 40 dots) that closes when `tool_started` fires. Both behaviours are suppressed for card-rendering tools (e.g. `edit_file`) so the card stays the single visual artifact for those calls.
- **`internal/permission` engine** — first slice of M6.5 (tool-safety + permission layer). YAML rule engine evaluates each `tool name × input` against an ordered list of `allow | deny | ask` rules; first match wins, no match = implicit ask. `~/.octo/permissions.yml` overrides the embedded `defaults.yml` per-tool. Default rules deny absolute disasters (`rm -rf /`, fork bombs, writes to `~/.ssh/**` / `/etc/**` / `**/.env*`, fetches to RFC1918 / `169.254.169.254` cloud-metadata), allow common safe commands (`ls`, `git status`, `go test`), and ask for everything else risky (`rm -rf`, `git push --force`, `sudo`, `curl`, outside-CWD writes). Two modes: `interactive` (CLI default — ask → prompt) and `strict` (M8 / M9 default — ask → deny + structured error so the LLM can explain to its upstream caller). Session-scoped `Remember` cache for "always allow this turn / session" answers.
- **Permission gating wired into the agent loop** — second slice of M6.5. `agent.PermissionGate` interface; `Agent.Gate` (nil = no gating, pre-M6.5 behaviour) is consulted in `dispatchTools` immediately before every tool call. A denied call never reaches the executor — the loop synthesises an `IsError` tool_result carrying the reason, so the LLM sees the denial and can adapt rather than the run aborting. The CLI builds a `cliPermissionGate` over the engine: `allow` passes through, `deny` returns the structured reason, `ask` (interactive only) prompts `[y]es / [a]lways this session / [N]o` on the shared REPL stdin (answering "always" caches the decision via `Remember`). New `octo chat --permission-mode {interactive|strict}` flag; an invalid value fails closed (exit 2) rather than silently dropping to the more-permissive mode. Gating is active whenever `--tools` is on.
- **Read-before-write enforcement** — third slice of M6.5. New `tools.ReadTracker` (one per session, held by `tools.NewDefaultRegistry()`) records every successful `read_file` / `write_file` / `edit_file` with the file's mtime. `write_file` / `edit_file` to an *existing* file is refused unless the file was read this session (`"File has not been read yet…"`) or if its mtime advanced out-of-band since the read (`"File has been modified since…"`) — mirroring Claude Code's wording so the LLM re-reads then retries. Creating a brand-new file needs no prior read; a write/edit re-stamps the tracker so a write→edit sequence doesn't demand a redundant read. The zero-value `DefaultRegistry{}` keeps the old no-enforcement behaviour for tests.
- **Terminal danger detection** — fourth slice of M6.5. (1) Credential-path access via `terminal` now asks even when the command verb is otherwise auto-allowed — `/etc/passwd`, `/etc/shadow`, `.ssh/id_`, `.aws/credentials`, `id_rsa`, `id_ed25519` patterns sit ahead of the safe-verb allows in `defaults.yml`, closing the hole where `allow: cat ` would let `cat /etc/passwd` slip through. (2) In-place stream edits (`sed -i`, `sed -i.bak`, `sed --in-place`, bundled `-ni`) are denied in `defaults.yml` and additionally guarded inside the terminal tool with a message steering the model to `edit_file` — so the change goes through permission + diff + read-before-write rather than bypassing them. The guard stops at shell separators (`| ; &`) so a non-in-place pipeline doesn't false-positive.
- **Tool input hardening** — final slice of M6.5, three independent guards. (1) `write_file` refuses content matching high-confidence secret shapes (PEM private-key blocks, GitHub `ghp_`/`github_pat_` tokens, `aws_secret_access_key` assignments, Slack `xox*` tokens); bare AWS access-key IDs like the docs `AKIAIOSFODNN7EXAMPLE` are deliberately *not* flagged to avoid false positives on tutorials. (2) `web_fetch` uses a dedicated HTTP client that refuses to follow a **cross-host** 3xx redirect (same-host path redirects still followed) — closes the SSRF/exfil vector where the proxy bounces the fetch to an unexpected host; `web_search` keeps the plain client since search backends legitimately redirect across hosts. (3) `read_file` rejects UNC / network paths (`\\server\share`, `//server/share`) before resolving, since on Windows they trigger an SMB connection that can leak NTLM credentials.
