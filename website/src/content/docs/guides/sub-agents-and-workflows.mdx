Two ways to get more than one agent working on a task: let the model spawn sub-agents ad hoc, or
write a deterministic orchestration script with the `workflow` tool.

## Sub-agents

The `sub_agent` tool spawns a fresh agent with its own context — useful for isolating a large
exploration or a risky edit from the parent conversation. Spawns can run synchronously or async;
async ones get follow-up tools:

| Tool | Purpose |
|---|---|
| `sub_agent` | spawn a sub-agent (sync or async, fork or specialized type) |
| `sub_agent_send` | send a follow-up message to a running or finished async sub-agent |
| `sub_agent_status` | check progress without blocking |
| `sub_agent_kill` | stop one early |

An async sub-agent that finishes without being killed stays addressable — a live handle you can
resume with `sub_agent_send` later in the conversation.

## Workflows

The `workflow` tool runs a deterministic orchestration script (JavaScript or an embedded Ruby DSL)
against a fleet of sub-agents, rather than leaving fan-out/fan-in to the model's judgment call by
call:

- **`agent(prompt, opts)`** — run one sub-agent to completion, return its result.
- **`parallel(fns)`** — run agents concurrently, wait for all.
- **`pipeline(items, ...stages)`** — run each item through stages independently, no barrier between
  stages — item A can be in stage 3 while item B is still in stage 1.
- Runs can go to the **background**: `workflow_status` polls, `workflow_kill` stops one.
- Each parallel branch that touches files gets **git worktree isolation**, so concurrent agents
  don't stomp on each other's working tree.
- `workflow_save` persists a script as a named, reusable workflow.

```js
// sketch: fan out a review across dimensions, then verify each finding
const findings = await parallel(
  ['correctness', 'security', 'perf'].map((d) => () => agent(`Review for ${d} issues`))
);
```
