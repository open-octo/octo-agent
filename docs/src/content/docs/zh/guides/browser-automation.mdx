octo 通过 DevTools Protocol（CDP）直接操作一个真实的 Chrome 标签页——不依赖任何 headless 浏览器
框架，也不需要额外的运行时。

```bash
octo browser setup
```

## 怎么连上 Chrome

`browser` 工具按下面这个顺序尝试拿到一个页面：

1. **已知的调试端口**（config.yml 里的 `browser.connect_port`）——直接连，失败了不会退而求其次。
   这就是 `octo browser setup` 帮你接好的那条路。
2. **按指定 profile 接入**（`browser.attach_running: true` 加上 `browser.user_data_dir`）——
   读取该 profile 目录下的 `DevToolsActivePort` 文件，连到拥有它的那个 Chrome。
3. **接入任何一个已开远程调试的 Chrome**（只配 `browser.attach_running: true`，什么都不配时
   也是这条默认路径）——自动发现一个开了调试的 Chrome，复用你的登录态；绝不会去劫持一个
   原本没开调试的浏览器。

没有"启动兜底"这回事：找不到可接入的 Chrome 时，工具返回的是一份打开远程调试开关的操作
指引，而不是悄悄启动一个没有任何登录态的临时实例。

不管走哪条路连上，octo 都会打开一个**全新标签页**，而不是复用你已经开着的某个标签页（包括它自己
Web UI 的那个）——要操作一个已有标签页，必须显式调用 `pages`/`select_page` 这类动作。连接会在整个
会话里复用，调试器授权提示只需要点一次，不会每次导航都弹。

### `octo browser setup`

```bash
octo browser setup
```

带你一步步打开 `chrome://inspect/#remote-debugging`（Chrome 144+ 之后默认 profile 上的旧
`--remote-debugging-port` flag 被禁用了，所以走 inspect 页面的勾选框是目前支持的路径），然后循环
探测 9222 端口——不只是确认连上了，还会验证一次真正的页面级 CDP 调用能成功（浏览器级连接可能成功，
但页面控制在新版 Chrome 上还是会失败）。成功之后会把 `browser.connect_port` 存进你的配置；失败就
提示你去开那个开关，回车重试，或者随时退出、以后再跑一遍。

## 录制

这个动作没有对应的 CLI 命令——录制是通过 `browser` 工具的 `record_start` / `record_stop` 两个动作
驱动的，可以由模型调用，也可以点 Web UI 的录制按钮。

关键的一点：**它录的是你做的操作，不是模型做的操作。** 调用 `record_start` 之后，agent 会把浏览器
交给你，由你亲自完成这些步骤，说一声"做完了"之后，agent 才调用 `record_stop <name>`。

每一步会记录：动作类型（`click` / `type` / `select` / `upload` / `navigate`）、一个锚定在最近的、
带稳定 `id` 的祖先元素上的选择器（而不是脆弱的、按位置算的选择器链）、这个元素的可见文本，以及
当时的 URL。之后会有一个模型处理步骤对原始录制做提炼——去掉绕路的死胡同、把你输入的具体值换成
`{{param}}` 占位符、写一段描述——但它只能重新排列或者重命名真实录到的步骤，不能凭空造一个新目标：
任何一步如果它的选择器不在原始录制里，就会被拒绝，退回用原始版本。

录制会以纯 YAML 的形式存在 `~/.octo/browser-recordings/<name>.yaml`——可读、可手改、可以用 git 追踪
差异。（改名前位于 `~/.octo/browser-skills/`，已有目录会自动迁移。）

## 回放

```
browser(action: "replay", name: "<录制名>", params: { ... })
```

（`run_skill` 作为已废弃的别名仍然可用。）

回放在常见情况下是确定性的——不涉及模型调用。每一步会先等目标出现，再执行，如果录制里声明了
`verify` 还会做一次校验。几个值得知道的健壮性细节：

- 点击会先找原始录制里的那个选择器；如果页面结构有点变化，但一个带着录制时记下的那段文字的元素
  还在，就会点那个元素，而不是直接失败。
- 如果某一步打开了新标签页，后续步骤会自动跟到那个新标签页上。
- 输入之后如果字段意外变成空的，会先清空重打一次，再失败才真正判定这一步失败。

返回结果是一个结构化对象——`{recording, steps, outputs, self_healed?}`——所以一段录制声明的
`outputs`（提取出来的值、下载文件的路径）可以通过 `skill("browser:<name>", params)` 直接接进
[工作流](/docs/zh/guides/workflows/)。

:::caution[回放是"要么整段做完，要么不做"]
octo 被明确指示：只有当请求和一段录制**从头到尾**完全匹配时才应该用它——回放会原样执行每一步，
不能只执行一部分，也不能在声明的参数之外做任何调整。如果请求在录制没有参数化的地方有出入，模型
会被引导直接操作浏览器，而不是硬套一段录制。这是 prompt 层面的指示，不是代码强制的限制——是在
录制曾经被过度信任、用于部分匹配的请求之后加上的。录制也是只能回放的：没有关键词触发这条路
（早期版本试过又撤了，不够可靠）。
:::

## 自愈

如果某一步的选择器找不到任何东西了——而且只有在配置了用于这个用途的模型时——octo 会对页面当前
可交互的元素取一份**文本摘要**（选择器 + 可见文本，不截图，不挑模型），把这份摘要连同预期动作、
预期的文字标签、失效的旧选择器一起发给模型，让它只回复一个修正后的 CSS 选择器。修正会被重试一次，
如果成功就写回这段录制的 YAML 文件——所以一次自愈是持久生效的，不是临时补丁，以后每次回放都不用
再修一遍。（写回会重新序列化整个 YAML，所以文件里手写的注释在下一次自愈回放后会被丢弃；字段值都会
保留。如果你会给录制加注释，建议用 git 管理它们。）

## 权限与视觉能力

浏览器动作走的是和其他工具一样的权限引擎——在一个已登录账号上点击或者提交表单，会被当成一次
真实的、影响很大的动作，而不是只读的一步。截图只有在当前模型支持视觉能力时才会以图片 block
返回；纯文本模型拿到的是一条文字提示，而不是一个会被拒绝的图片 block。
