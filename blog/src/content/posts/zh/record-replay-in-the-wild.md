# Record & Replay 在真实网站上活下来之前，我们修了什么

## 开篇：一个"半小时能搞定"的 demo

需求听起来一句话就说完了：给 octo 的浏览器 Record & Replay 录一段真实 demo——我亲手在网站上发一篇文章，octo 在旁边录；然后换一篇文章，让它自己从头到尾回放一遍。

两个真实网站（知乎专栏、小红书创作平台），最后变成了二十多个提交、横跨录制器、编译器、回放引擎、自愈系统四层的修复。**每一次失败都不是我们代码里的低级错误，而是现代 Web 的某个真实机制在和自动化作对。** 更值得写下来的是最后那一段——我们打了一连串各自都正确、都有测试的点击修复，却发现它们在追一个幻影，真正的病灶要靠取证才能揪出来。

这篇文章按发现顺序复盘。如果你在做浏览器自动化——不管用不用 octo——这份清单大概率能帮你省掉几天。

## 第一坑：点了"写文章"，录制就聋了

**症状**：在知乎录制发文章。点导航栏"写文章"跳到编辑器后，后面所有操作一步都没录上。录制文件在新标签页打开的那一刻戛然而止。

**根因**：知乎的编辑器开在新标签页里，而这个新标签页是 `noopener` 的。我们的录制器用 CDP 的 `Target.setAutoAttach`（freeze-on-create）监听新 target——但这个调用挂在**被录制页面的 session** 上，session 级 auto-attach 只对"有 opener 关系的相关 target"生效。关键事实是：**Chrome 88 起，`target="_blank"` 默认隐含 `rel="noopener"`**。也就是说，网页上最普通的"新标签页打开"链接，创建的都是无关联的独立 target，`Target.attachedToTarget` 根本不会触发。录制器不是漏了新标签页的处理，是它的处理只覆盖了 `window.open` 保留 opener 的那一小类（OAuth 弹窗）。

**修复**：录制期间额外申请一个 **browser 级**的 freeze-on-create auto-attach（`sessionId` 为空的那种），它对所有新建 target 生效，不看 opener。录制结束时关掉。

## 第二坑：冻结的标签页不跟你说话

修完第一坑，新标签页能 attach 了，却发现注入监听脚本的调用全部超时。

**根因**：freeze-on-create 把新标签页冻结在创建瞬间，等注入完成再恢复——这个设计本身是对的（防止用户手快，页面还没被监听就被点了）。但有 opener 的弹窗共享打开者的渲染进程，冻结期间照常应答 CDP 命令；**noopener 标签页的新渲染进程在恢复之前不应答任何命令**。于是"同步调用注入命令、等返回、再恢复"的写法和自己的恢复调用互相死锁。

**修复**：Puppeteer 同款模式——把整套注入命令**先后写入 socket 但不等响应**，紧接着写入恢复命令，事后统一收响应。CDP 的 wire 顺序等于程序顺序，排队的注入保证在页面任何代码执行之前生效，冻结的防抢跑保证原样保留。

顺手还发现第三个坑：一个标签页可能同时存在**多条 CDP session**（auto-attach 一条、显式 attach 一条），而页面里一次 binding 调用会通知**所有**注册过该 binding 的 session——每个动作录两遍。去重的 key 必须是 target，不能是 session。

## 第四坑：自愈在数学上不可能成功

**症状**：回放失败，报 "no element matches the recorded fingerprint"。但我们明明做了 LLM 自愈——为什么没修？

排查后发现两层问题。第一层是**可观测性**：自愈其实跑了，模型也给出了答案，但失败时代码把原始错误原样返回——"自愈失败"和"自愈从没跑过"在错误信息上一模一样。第二层更有意思：**自愈成功了也没用**。healer 只改选择器，重试时带指纹的步骤又要过同一道指纹评分门槛——而指纹（录制时的邻居文本、role、tag）正是刚刚匹配失败的那个东西。指纹本身漂移的场景下，自愈在数学上不可能通过。

**修复**：healer 给出的修复选择器是**权威的**——重试直接用它，过期指纹随修复一起丢弃并写回文件；自愈的每个出口都在错误信息里自报家门（`self-heal gave up: …`、`tried 3 rounds without success`）。

## 第五坑：一个 `~` 杀死了整个页面

**症状**：回放到上传文件那步，Chrome 弹出"喔唷，崩溃啦！`RESULT_CODE_KILLED_BAD_MESSAGE`"，然后整个回放挂死了 400 秒才超时。

**根因**：上传参数里的 `~/Downloads/…` 被原样传给了 `DOM.setFileInputFiles`。Chrome 对浏览器进程无法授权的路径（未展开的 `~`、相对路径、不存在的文件）的反应不是报错，是**直接杀掉渲染进程**——它认为渲染进程被攻破了。更糟的是连锁反应：死掉的渲染进程让所有 CDP 命令永远等不到响应，而每一步的等待超时**本身也是 CDP 轮询实现的**，跟着一起挂。

**修复**：两层。路径在进 Chrome 之前归一化（展开 `~`、必须绝对路径、检查存在性），把"杀进程"变成一条干净的步骤错误；回放的每一步套一个硬超时兜底（步骤超时的 8 倍、导航超时的 2 倍取大），死掉的标签页最多拖两分钟，不再拖垮整轮对话。

## 第六坑：页面自己点了自己

**症状**：换到小红书。连续三次录制，上传步骤都编译成了一个不可回放的裸 `input` 选择器，而真正被点击的可见上传区域成了孤立的一步。

这是全程最绕的一个坑，破案靠的是我们中途加的**原始事件侧车**（`events.json`，record_stop 时把未加工的捕获事件存在编译产物旁边）。事件流里赫然多出一次没人做过的点击：

```
13 click  DIV    …upload-area          ← 用户真实点击上传区域
14 click  INPUT  input                 ← ???
15 upload INPUT  input
```

**根因**：小红书的上传区域被点击后，页面 JS **动态创建一个 file input 并程序化调用 `.click()`**。这个合成点击的 `isTrusted` 是 `false`，但录制器照单全收。后续的"点击+选文件合并成单个上传步骤"逻辑向前回看时撞上的是这个假点击，把上传目标合并成了裸 `input`。

**修复**：录制只接受 `isTrusted` 的 click 和 Enter。程序化事件是页面的内部实现，不是用户的演示。（change 事件保持不过滤——select 模拟和部分框架控件合法地派发合成 change。）

## 第七坑：SPA 的回声把状态冲掉了

**症状**：回放到"新的创作"按钮时报找不到元素。截图显示页面停在默认的"上传视频"标签，而不是录制时的"写长文"标签。

**根因**：点"写长文"是一次 SPA 标签切换，页面用 `pushState` 重新宣告了一次自己的 URL（加了个 `&from=tab_switch`）。录制器把它录成了一个 navigate 步骤——回放执行这个 navigate 时**整页重载**，刚被点出来的标签选中态被冲回默认值。

修这个坑经历了两次迭代。第一版是回放侧打补丁：目标 URL 和当前位置相同就跳过重载，再给 SPA 的异步 URL 更新留一个同 host 的宽限窗口。能救一部分场景，但治标——回放时页面未必会附加同样的 query 参数，等不到就还是会强制加载。第二版才是语义级的答案：**`navigatedWithinDocument`（pushState/replaceState）只可能由页面 JS 发起**——地址栏输入永远触发整页加载。所以录到的一切"文档内导航"都是前一个动作的*效果*而非用户*动作*，回放靠重放那个点击即可复现，编译时应整类丢弃。已知的代价只有一个：SPA 页面上的浏览器前进/后退按钮（本来也没有可捕获的 DOM 手势）。

## 第八坑：没有语义的步骤，谁都救不了

贯穿多次失败的一个底层问题：录制器捕获的目标元素经常**没有任何语义**——用户点的是按钮里的 svg 图标、portal 弹层里的裸 div，录下来的步骤是 `click div:nth-of-type(4)`，没有文本、没有 role、没有 hint。这种步骤页面一变就死，而且自愈也无能为力：模型拿到的全部线索就是"点一个 div"。

**修复**是一组：点击**重定位到最近的可交互祖先**（button/a/[role=button|menuitem|…]）再生成选择器和指纹——语义都在祖先身上；选择器生成器把**易变 id**（`react-aria-N`、`PopoverN-…`、`:rN:` 这类框架自增 id，跨会话必变）降级，优先 aria-label/name/placeholder；中文表单提示不再被 slug 成空串跌进 `value` 这种兜底名——`{{输入标题}}` 比 `{{value}}` 自解释得多，模型也不用猜。

参数这条还有个配套教训：record_stop 的回执原本不列参数清单，模型向用户复述计划时会**凭语感编一个参数名**（把 `value` 说成 `title`），然后回放时照抄自己编的。修复就是把声明的参数原样列进工具结果——需要判断的地方，把证据喂足。

## 第九坑：回放比人快，快到踩进了人类没见过的状态

前八坑修完，回放能稳定跑到发布页了：登录态、写长文、导入文档、填标题、正文、一键排版、下一步——全过。只剩最后一下：点"发布"。然后它开始了一场旷日持久的拉锯——**18 步全绿，小红书笔记管理里却什么都没有**。

发布按钮 `<xhs-publish-btn>` 是个 **closed Shadow DOM 自定义元素**：页面上下文看不进它内部，`shadowRoot` 返回 null，`childElementCount` 恒为 0，任何选择器都穿不进去找到里面的按钮。这类控件只有一个可靠信号能判断"就绪"——标准的 `:defined` 伪类（类升级完才匹配）；也只有一种点法——对宿主用坐标点。

围绕这个按钮，我们接连补了一整套**点击 actionability**，每一条都是真实且通用的加固，每一条都有回归测试：

- **坐标保真**：录制记下用户按压点在元素盒内的相对比例（`click_x`/`click_y`），回放点同一处而非几何中心——closed shadow 宿主的中心可能是死区。
- **就绪等待**：点击前等 `document.readyState === complete`、等自定义元素 `:defined`、等自身及近祖先退出 `disabled` / `*-loading` 态。
- **命中测试**：点击点必须 `document.elementFromPoint` 真能打到目标——转场遮罩盖住时命中的是遮罩，于是等它散去（这招能穿透 closed shadow）。
- **真实手势**：先把指针 `mouseMoved` 到目标、停一帧多让 pointer-entry 处理器武装，再 `mousePressed(buttons:1)`——裸的按下/抬起对靠 pointer 进入才激活的组件是无效的。

每修一条，回放都往前挪一点，但发布点击**还是**时灵时不灵。这里藏着录制回放这类系统特有的一类病：**回放比任何人都快，快到踩进了人类从未见过的中间状态**——组件壳出现了但没水合、页面转场还没完、遮罩还在转圈。上面每一条都在堵这类窗口。

## 第十坑：wait-for-idle 在请求发出之前就返回了

用户的一句观察把病灶又往上游推了一层：点完"下一步"，页面还没变、"下一步"按钮自己还在转圈，第 18 步（发布）**却已经发生了**。中间那步"等网络空闲"秒过了。

**根因**：点击触发的提交请求还没发出的那个空隙里，网络监视器看到"当前 n=0、从页面加载起就空闲"，立刻判定 idle 返回。于是后续步骤在页面还在转场时就冲了出去。这是"等空闲"的经典竞态：**它没先确认活动发生过，就宣布了空闲。**

**修复**：`WaitForNetworkIdle` 先读一个单调递增的"活动代数"，**给活动一个有界宽限去出现**（代数变了、或抓到 busy），再等它平息；真没有活动才走快速路径返回。这条修完，回放才真正能等到每一次 SPA 转场完成——它本该桥接整条流程的每个转场，之前结构上根本等不住。

## 第十一坑：一串各自正确的修复，其实在追一个幻影

到这里，回放稳定跑到发布页、点击 actionability 齐全、转场等待可靠——发布点击**还是**没生效。我们已经为这一下打了三四轮补丁，每个单独看都对、都有测试，但它总能换个花样失效。

我停下来，做了本该早点做的事：**加一段临时诊断日志**，把回放实际解析到哪个元素、点到什么坐标、那个坐标 `elementFromPoint` 命中谁、是否就绪，全部打到日志里，然后请用户再回放一次。数据一锤定音：

```
selector=… > xhs-publish-btn  point=(649,791)
target=XHS-PUBLISH-BTN  targetRect=[229,749,680,90]
hitAtPoint=XHS-PUBLISH-BTN  sameSubtree=true  defined=true
```

元素对、坐标在宿主内、命中宿主、已就绪——**每一个闸门都通过了，点还是没生效**。与此同时用户看到了一个之前没提的细节：页面弹了一句 **"图片正在上传"**。

真相在这里翻转：**点击一直是对的**。发布是小红书长文的一个异步前置条件——平台自动生成的封面图还在上传，服务端**校验**拦下了发布。我们那一整套点击 actionability 修复，对这个 case 而言是在**追一个幻影**——它们是真实且通用的加固（每个都有测试、每个都让别的场景更稳），但本案的堵点根本不在点击层，而在服务端。

这一坑最该记住的不是某个补丁，而是方法：**取证比猜快**。我们凭"点太快/点不到"的直觉连打数轮，直到一段诊断日志 + 用户的一句"图片正在上传"，才把方向从客户端拽到服务端。

**修复**：给"决定性点击"（录制里其后紧跟终点、`end_url` 已知、本意是跳转的那个点击）加一层重试——效果未达成时**等网络平息、再点**，直到 URL 真的前进或用尽次数。关键是它用 **URL 前进** 判定成功，而不是网络活动——真实页面的后台心跳、埋点会让"有没有网络活动"永远为真（这正是更早一版基于网络的重试没救成的原因）。发布被"图片正在上传"挡回 → 等封面传完 → 再点 → URL 跳到 `?published=true`。

## 第十二坑：回放不知道自己失败了

第十一坑还暴露了一个结构性问题：发布点击哑掉之后，**后面的等待步骤照样通过**——网络空闲的等待在"什么都没发生"的页面上瞬间满足。回放报告全部成功，实际上什么都没发出去。

**修复**：录制记录**演示终点 URL**（`end_url`——包括那些被正确丢弃、从不编译成步骤的 pushState 重定向，比如 `…published=true`）。回放结果里并列报告 `demonstrated_end_url` 和 `replay_end_url`，不一致时附带明确警告：先去页面核实结果，再报告成功。终态不符不算回放失败（query 参数可以合法变化），但它把"沉默的假成功"变成了一个模型必须处理的信号。这一环上线后的第一次回放就见效了——octo 的回复从"回放完成 ✅"变成了"回放完成，但有一个需要确认的问题"。

## 方法论：证据侧车与自主修复

把这些坑排完，留下来的不只是修复，还有一套可观测性基建，它们本身就是排查方法论的物化：

- **`events.json` 侧车**——每次录制把原始捕获事件存在编译产物旁边。编译是有损的蒸馏，出问题时侧车是唯一能区分三类故障的证据：**事件里没有 → 捕获缺口，重录；事件里有但步骤不对 → 编译错误，改 YAML；两边都对 → 页面变了，修选择器加等待**。第六坑就是它破的案。
- **逐步进度流**——回放不再是一个几十秒的黑盒，每一步开始、每次自愈介入都实时上报到界面和 CLI。
- **失败自报家门**——错误信息带自愈轨迹、带诊断文件路径、带一棵决策树；碰到"图片正在上传"这类瞬时校验、碰到 closed shadow 控件，都明确告诉模型该等一下重放、还是别用文本去点。

最后这点值得展开：这些证据不只是给我们看的。octo 的回放失败信息会把 YAML 路径、events.json 路径和决策树一起交给模型——**引擎修不了的问题需要判断，需要判断的问题需要证据**。用户的 octo 在录制回放失败时，拿到的排查装备和我们开发时是同一套。

## 尾声：给浏览器自动化作者的敌意清单

最后那次回放，octo 完全自主生成的录制、自主回放，文章真的发出去了。回头看，这一路的坑没有一个是我们代码里的笔误，全部是现代 Web 的真实机制：

1. `target="_blank"` 默认 noopener（Chrome 88+），session 级 auto-attach 看不见
2. noopener 新标签页的渲染进程在恢复前不应答 CDP 命令
3. 一个 target 可以有多条 session，binding 调用全体广播
4. `DOM.setFileInputFiles` 收到不可授权路径会杀渲染进程，而不是报错
5. 死掉的渲染进程让 CDP 调用无限挂起——你的超时机制可能也是 CDP 实现的
6. 页面会程序化地点击自己（`isTrusted=false` 是唯一的分界线）
7. SPA 的 `pushState` 会重播当前 URL——录下它，回放就是一次整页重载
8. 框架自增 id（`react-aria-N`）跨会话必变
9. closed Shadow DOM 把一切事件重定向到宿主，宿主中心可能是死区，`childCount` 恒为 0，`:defined` 是唯一就绪信号
10. 组件渲染出来 ≠ 组件醒了：`*-disabled`/`*-loading`、未水合、遮罩未散期间的点击被静默吞掉，而回放永远比人快
11. 裸的按下/抬起对靠 pointer 进入才激活的组件无效——点前要先移动、停一帧
12. "等网络空闲"若不先确认活动发生过，会在请求发出前的空隙里秒返回
13. 决定性动作会被服务端瞬时校验挡回（"图片正在上传"），而这只能靠"效果是否达成"判断，不能靠"有没有网络活动"——真实页面的心跳会骗过后者

还有一条方法论的，比上面任何一条都重要：**当一串各自正确、各自有测试的修复仍然修不好同一个症状时，停下来取证，别再猜。** 我们追了三四轮点击时序的幻影，真正的病灶——服务端的一句"图片正在上传"——是一段诊断日志加用户的一句观察揪出来的。

（附一条测试环境的坑：headless Chrome 会节流 JS 定时器，用 `setInterval` 驱动的动画在满载的 CI runner 上会被卡成"不动"，让依赖"元素在移动"的测试假过——改用 CSS transition，渲染引擎驱动，节流不动它。）

octo-agent 的这套 Record & Replay 是开源的（`internal/browser/`），上面每个修复都带着一个用最小合成页面写成的回归测试。如果你踩到了清单之外的坑，欢迎来提 issue——这份清单显然还没写完。
